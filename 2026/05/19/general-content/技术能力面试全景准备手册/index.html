一、架构方法论

1.1 DDD 领域驱动设计

核心知识

DDD（Domain-Driven Design）是一套以业务领域为核心的软件设计方法论，核心分为战略设计和战术设计两层。

战略设计

  • 限界上下文（Bounded Context）：业务边界划分的核心工具，每个上下文有独立的语言模型（Ubiquitous Language），上下文之间通过防腐层（ACL）、共享内核（Shared Kernel）、发布/订阅等方式集成
  • 上下文映射（Context Map）：描述上下文间关系，包括 Upstream/Downstream、Conformist、Customer/Supplier 等模式
  • 核心域 / 支撑域 / 通用域：按业务价值分类，指导投入重点

战术设计（构建块）

  • 实体（Entity）：有唯一标识，生命周期内状态可变（如订单 OrderId）
  • 值对象（Value Object）：无标识，通过属性描述，不可变（如货币金额 Money）
  • 聚合（Aggregate）：一组强一致性的实体+值对象，聚合根（Aggregate Root）作为唯一入口，保证内部不变量
  • 领域事件（Domain Event）：领域中发生的业务事实，是跨聚合、跨上下文协作的核心机制
  • 仓储（Repository）：聚合的持久化抽象，隔离领域层与基础设施层
  • 领域服务（Domain Service）：无法归属单一实体的领域逻辑，通常是跨实体的操作
  • 应用服务（Application Service）：用例编排层，事务边界管理，不含业务规则

分层架构（经典四层）

应用场景

1.2 微服务分层架构

核心知识

微服务分层架构定义了服务内部和服务间的职责边界，通常包含以下层次：

服务内分层（典型 Spring Boot 项目）

服务间分层

  • API 网关层：统一入口，负责路由、认证鉴权、限流熔断、协议转换（Kong/Nginx）
  • BFF 层（Backend for Frontend）：为特定客户端聚合多服务数据，减少前端请求次数
  • 业务服务层：按限界上下文拆分，服务间通过 RPC 或消息通信
  • 基础服务层：用户中心、权限服务、配置中心等横切关注点
  • 数据层：每个服务独占数据库（Database per Service 模式）

服务通信模式

  • 同步：REST / Thrift RPC / gRPC
  • 异步：消息队列（MQ 解耦、削峰）
  • 服务发现：Nacos / Eureka（客户端发现 vs 服务端发现）

应用场景

  • 车联网平台：设备接入服务、数据解析服务、业务处理服务、推送服务分层，各层独立扩缩容
  • 电商大促：API 网关限流保护下游，BFF 聚合商品+库存+促销数据减少前端轮询

高频面试问答

Q1：微服务拆分粒度如何把握？拆太细有什么问题？

A：拆分粒度应以业务能力而非技术为边界，遵循高内聚低耦合原则。
拆太细的问题：

  • 分布式调用链路过长，延迟累积，故障定位难
  • 运维开销指数级增加（每个服务独立部署、监控、日志）
  • 分布式事务复杂度上升

建议：优先按限界上下文划分，初期可以”宽服务”起步，待业务边界清晰后再细拆。通常一个服务对应一个2-pizza团队（6-8人）管理范围。

Q2：服务间如何处理接口版本兼容？

A：常见策略：

  1. URL 版本号：/api/v1/orders，简单直观但 URL 冗余
  2. Header 版本：Accept: application/vnd.app.v2+json，语义更 RESTful
  3. 向后兼容演进：新增字段不破坏旧客户端，删字段走废弃流程+版本协商
  4. 消费者驱动契约测试（CDC）：Pact 框架验证服务间契约，在 CI 中捕获不兼容变更

1.3 CQRS（命令查询职责分离）

核心知识

CQRS 将系统操作分为两类：

  • Command（命令）：修改状态，不返回数据（或仅返回操作结果标识）
  • Query（查询）：读取状态，不产生副作用

核心思想：读写模型分离，分别优化

典型实现

  • 写模型：DDD 聚合，保证强一致性
  • 读模型：专为查询优化的扁平化数据结构（如宽表、Elasticsearch 索引）
  • 同步机制：写模型发布领域事件 → 事件处理器更新读模型（最终一致性）

与 Event Sourcing 的关系：CQRS 常与 ES 结合，写模型存储事件序列，读模型通过重放事件构建。两者独立，CQRS 不强制 ES。

应用场景

1.4 Saga 分布式事务

核心知识

Saga 是解决微服务间长事务问题的模式，将分布式事务拆分为一系列本地事务，每个本地事务发布事件/消息触发下一步，失败时执行**补偿事务（Compensating Transaction）**回滚。

两种协调模式

① 编排（Choreography）

  • 优点：服务解耦，无单点
  • 缺点：事务流程分散，难以追踪全局状态

② 编制（Orchestration）

  • 优点：流程集中管理，状态机清晰，易于监控
  • 缺点：Orchestrator 成为中心依赖，需处理幂等

关键设计要素

  • 幂等性：每个本地事务需支持幂等，防止消息重投导致重复执行
  • 补偿事务设计：补偿不是物理回滚，而是业务层面的反向操作（已发货无法撤销，只能发起退货）
  • 隔离性问题：Saga 只保证最终一致性，不提供传统 ACID 的隔离性，需防止”脏读”（通过语义锁、乐观锁等手段）

应用场景

  • 电商下单：创建订单 → 扣减库存 → 扣款 → 通知物流，任一步骤失败触发链式补偿
  • 车辆 OTA 升级：推送任务创建 → 设备下载 → 设备升级 → 结果上报，失败回滚升级状态

高频面试问答

Q1：Saga 如何保证幂等？消息重复投递怎么处理？

A：幂等保证有三种常用方式：

  1. 唯一事务ID + 状态表：每个 Saga 步骤携带唯一 txId，执行前查状态表，已执行则跳过
  2. 数据库唯一键防重：利用 DB 唯一索引，重复插入直接忽略
  3. 消息去重表：消息消费时写 msgId 到去重表，重复消息直接 ACK 不处理

RocketMQ 事务消息与 Saga 结合时，本地事务和消息发送是原子的，避免了本地成功但消息未发的问题。

Q2：编排模式和编制模式如何选择？

A：

  • 编排：适合步骤较少（≤3步）、团队自治要求高的场景，每个服务独立演进
  • 编制：适合流程复杂（多步骤/条件分支）、需要集中监控和重试策略的场景，如订单中台

实践中，编制模式 更容易实现”可观测性”——可以通过 Orchestrator 的状态机直观看到每笔事务卡在哪一步，便于运维排查。Seata 框架支持 Saga 编制模式。

1.5 高并发状态机

核心知识

状态机（FSM/HSM）用于建模具有明确状态转换的业务实体，在高并发场景下需要特别处理并发状态冲突。

核心要素

  • 状态（State）：实体所处的阶段（如订单：待支付/已支付/已发货/已完成/已取消）
  • 事件（Event）：触发状态转换的输入（如：支付成功事件）
  • 转换（Transition）：状态 + 事件 → 新状态
  • 动作（Action）：转换发生时执行的副作用

高并发下的并发控制

  • 乐观锁：通过 version 字段 CAS，适合冲突低的场景
  • 数据库行锁（SELECT FOR UPDATE）：适合冲突率高的场景，但要控制锁持有时间
  • 分布式锁（Redis）：跨节点状态变更时，先获取锁再执行状态转换
  • 消息队列串行化：将同一实体的状态变更事件入同一分区，单线程顺序消费，从根本上消除并发

状态机框架

  • Spring State Machine：适合复杂状态机，支持嵌套状态、并发区域
  • Stateless4j：轻量级，适合简单状态流转

应用场景

  • 订单状态流转：防止重复支付、重复发货
  • OTA 升级任务：任务状态流转（待推送→推送中→下载中→升级中→成功/失败）需要幂等，设备断联重连后状态恢复
  • IoT 设备在线状态：百万级设备并发上下线，状态机 + Redis 原子操作维护在线状态

高频面试问答

Q1：在分布式环境中如何保证状态机的状态一致性？

A：核心是将状态变更操作设计为原子操作：

  1. 数据库 CAS：UPDATE ... WHERE status = 'expected_status'，影响行数为0则表示冲突，返回失败让上层重试
  2. Redis Lua 脚本：Lua 脚本原子执行”读状态-判断-写状态”，避免 TOCTOU 问题
  3. Actor 模型（如 Akka）：每个业务实体对应一个 Actor，单线程处理消息，天然无并发
  4. 消息分区：同一设备/订单的消息路由到同一 Kafka 分区，消费者单线程保证顺序

Q2：状态机的非法状态转换如何防御？

A：在状态机层实现防御性编程：

  • 定义合法转换矩阵，收到非法事件时记录告警并拒绝（而非抛异常导致服务崩溃）
  • 数据库约束作为兜底（如 CHECK 约束）
  • 事件溯源（Event Sourcing）让所有历史变更可审计，便于排查非法状态出现的根因

1.6 聚合层服务编排

核心知识

聚合层（Aggregation Layer）/ 服务编排（Service Orchestration）位于业务服务之上，负责：

  • 数据聚合：组合多个下游服务的数据，减少前端/客户端请求数
  • 业务编排：串联多服务调用，实现复杂业务流程
  • 协议适配：将内部微服务协议转换为外部 API 协议

编排模式

关键技术

  • 并行调用：无依赖关系的下游调用并行执行（CompletableFuture.allOf / Reactor）
  • 超时熔断：Sentinel / Resilience4j 设置熔断，防止慢服务拖垮编排层
  • 数据裁剪：按需返回字段（GraphQL 天然支持；REST BFF 手动裁剪）
  • 缓存策略：聚合层缓存热点数据，降低下游压力

高频面试问答

Q1：服务编排中如何处理部分失败？

A：取决于业务语义：

  • 强依赖：任一服务失败则整体失败，返回错误（如支付必须成功）
  • 弱依赖：允许降级，用默认值填充（如推荐服务超时，返回空列表而不影响主流程）
  • 重试策略：指数退避重试，配合幂等保证

具体实现：Resilience4j 的 @CircuitBreaker + fallback 方法，定义降级兜底逻辑。

二、消息与通信

2.1 MQTT / EMQX（百万级长连接）

核心知识

MQTT 协议要点

  • 基于发布/订阅模型，TCP 长连接，专为低带宽、不稳定网络设计（物联网场景）
  • QoS 等级：
    • QoS 0：最多一次（At most once），无确认，可能丢失
    • QoS 1：至少一次（At least once），有 PUBACK，可能重复
    • QoS 2：精确一次（Exactly once），四次握手，性能最低
  • 保留消息（Retained Message）：新订阅者立即收到最后一条消息，用于设备状态初始化
  • 遗嘱消息（Last Will）：连接异常断开时 Broker 自动发布，用于设备离线通知
  • 持久会话（Clean Session=false）：客户端断线重连后，Broker 补发离线期间消息

EMQX 架构（百万级连接）

  • 基于 Erlang/OTP，Actor 模型 + 轻量进程，单节点支持百万连接
  • 集群方案：Mnesia 分布式数据库做节点间路由同步；Ekka 库管理集群成员
  • 规则引擎（Rule Engine）：消息到达后触发 SQL 规则，将数据路由到 Kafka/ClickHouse/DB 等后端
  • 共享订阅：多个消费者共享同一 Topic 订阅，实现负载均衡（$share/group/topic）
  • 认证鉴权：支持 JWT、X.509 证书、HTTP Hook 等多种认证方式

性能调优关键点

  • 调整 Erlang VM 参数：+P 最大进程数，+Q 消息队列长度
  • 操作系统调优：ulimit -n（文件描述符），TCP backlog，内核网络参数
  • 消息持久化会显著降低吞吐量，按场景选择 QoS 等级

应用场景

2.2 RocketMQ（事务消息）

核心知识

RocketMQ 核心概念

  • NameServer：无状态路由注册中心，Broker 向其注册，Producer/Consumer 从其获取路由（类似轻量 ZooKeeper，但无状态，宕机不影响已连接的 Producer/Consumer）
  • Broker：消息存储与转发，Master-Slave 主从复制，支持同步/异步双写
  • Topic/Queue：Topic 可分为多个 MessageQueue，分布在不同 Broker，实现横向扩展

消息存储机制

  • CommitLog：所有消息顺序写入，单文件 1GB，顺序 IO 性能极高
  • ConsumeQueue：每个 Topic-Queue 对应逻辑索引文件，存储 CommitLog offset + 消息大小
  • IndexFile：按 MessageKey 的哈希索引，支持消息查询

事务消息原理（两阶段 + 补偿）

这解决了”本地事务与消息发送的原子性”问题。

消费模式

  • 推模式（Push）：Broker 主动推，实际是长轮询模拟
  • 拉模式（Pull）：Consumer 主动拉，自控消费速率
  • 广播消费：每个 Consumer 实例都消费全量消息
  • 集群消费：消息被 Consumer Group 中的一个实例消费（负载均衡）

消息积压处理

  • 扩充 Consumer 实例（不超过 Queue 数量）
  • 临时提高消费线程数（consumeThreadMax）
  • 消息跳过（特殊情况下手动重置 offset）

高频面试问答

Q1：RocketMQ 事务消息如何解决分布式事务？与本地事务如何绑定？

A：事务消息实现”本地事务 + 消息发送”的最终原子性：

关键点：本地事务状态需要持久化（写 DB），供 Broker 回查时判断。

Q2：RocketMQ 如何保证消息不丢失？从生产到消费全链路分析。

A：三个环节分别保证：

  1. 生产端：同步发送 + 失败重试；开启事务消息（关键业务）；Broker 同步双写（syncFlush=true，SYNC_MASTER）
  2. Broker 端：CommitLog 同步刷盘（flushDiskType=SYNC_FLUSH）；主从同步复制
  3. 消费端：消费成功后再 ACK；死信队列（DLQ）兜底；消息消费失败重试（默认重试16次，每次间隔递增）

代价：同步刷盘 + 同步复制性能下降约30%，按业务重要性权衡配置。

Q3：RocketMQ 和 Kafka 如何选择？

A：

维度 RocketMQ Kafka
事务消息 原生支持 不支持（需自实现）
消费失败重试 内置重试+死信队列 需自实现
延迟消息 支持固定延迟级别 不支持
消息查询 支持 MessageKey 查询 不支持
吞吐量 较高（百万TPS） 极高（千万TPS）
顺序消息 队列级有序 分区级有序

选 RocketMQ：业务消息、事务、复杂消费语义
选 Kafka：日志收集、高吞吐流处理、数据管道

2.3 Kafka（高吞吐流处理）

核心知识

Kafka 高性能原理

  • 顺序写 + PageCache：消息追加写到日志文件，OS PageCache 吸收写 IO，吞吐量极高
  • 零拷贝（sendfile）：消息从磁盘到网卡不经用户态，减少4次数据拷贝为2次
  • 批量压缩：Producer 批量发送（linger.ms + batch.size），支持 gzip/snappy/lz4 压缩
  • 分区并行：Topic 分区分散在多 Broker，Producer/Consumer 并行读写

核心配置关键参数

  • 生产端：acks=all（所有副本确认）, retries, batch.size, linger.ms
  • 消费端：enable.auto.commit=false（手动提交 offset）, max.poll.records, isolation.level
  • Broker：min.insync.replicas=2，unclean.leader.election.enable=false

Kafka Streams / Flink on Kafka

  • Kafka Streams：轻量级流处理库，无需独立集群，状态存 RocksDB
  • 与 Flink 结合：Kafka 作为 Source/Sink，Flink 做复杂流处理（窗口、Join、CEP）

分区策略与消费者组

  • 同一 Consumer Group，每个 Partition 只被一个 Consumer 消费
  • Consumer 数 > Partition 数：多余 Consumer 空闲
  • 扩容：增加 Partition 数，触发 Rebalance（有短暂停顿，可用 CooperativeSticky 分配策略减少影响）

高频面试问答

Q1：Kafka 如何保证消息 Exactly-Once 语义？

A：Kafka 通过两个机制组合实现 Exactly-Once：

  1. 幂等生产者（Idempotent Producer）：Producer 分配 PID + SequenceNumber，Broker 检测重复并去重，解决 Producer 重试导致的重复写入
  2. 事务（Transactions）：producer.beginTransaction() → 发送到多个 Partition → producer.commitTransaction()，消费端设置 isolation.level=read_committed 只读已提交消息

端到端 EOS 需要：幂等 Producer + 事务 + 消费者提交 offset 在同一事务内

Q2：Kafka Rebalance 的触发条件和影响，如何优化？

A：触发条件：Consumer 加入/离开 Group；Consumer 超时（session.timeout.ms）；订阅 Topic 分区数变化。
影响：Rebalance 期间所有 Consumer 暂停消费（Stop-the-World），可能导致消费延迟。
优化：

  • 增大 session.timeout.ms 和 heartbeat.interval.ms，减少误判退出
  • 使用 Incremental Cooperative Rebalancing（Kafka 2.4+，partition.assignment.strategy=CooperativeStickyAssignor），只迁移必要分区，其余正常消费
  • 避免频繁扩缩 Consumer 实例

  2. 新 Pod IP 注册到 Endpoints

4.1 MySQL

核心知识

InnoDB 存储引擎关键机制

  • B+ Tree 索引：叶节点存完整行数据（聚簇索引）；非叶节点只存键值；二级索引叶节点存主键，需回表查询
  • 事务 ACID 实现：
    • 原子性：Undo Log（回滚）
    • 持久性：Redo Log（WAL，崩溃恢复）
    • 隔离性：MVCC + 锁
    • 一致性：约束 + 上述机制组合
  • MVCC：每行数据有 trx_id 和 roll_pointer，读操作通过 ReadView 判断可见版本，实现无锁读
  • 锁机制：行锁（Record Lock）、间隙锁（Gap Lock）、临键锁（Next-Key Lock，防幻读）
  • Buffer Pool：LRU 链表管理，Young/Old 区防止全表扫描污染热数据

索引优化要点

  • 覆盖索引：查询字段全在索引中，无需回表
  • 最左前缀原则：联合索引从左匹配
  • 索引失效场景：函数运算、隐式类型转换、OR 连接、LIKE '%前缀'、NOT IN

主从复制

  • 基于 Binlog（ROW 格式更安全），异步/半同步/MGR（Group Replication）
  • GTID（全局事务ID）简化故障切换，避免 Binlog Position 对齐

高频面试问答

Q1：MVCC 如何解决幻读？可重复读（RR）隔离级别下幻读还存在吗？

A：MySQL RR 通过 MVCC + Next-Key Lock 解决幻读：

  • 快照读（SELECT）：MVCC 读事务开始时的快照，同一事务内多次读结果一致，天然无幻读
  • 当前读（SELECT FOR UPDATE / UPDATE）：加 Next-Key Lock 锁住查询范围+间隙，阻止其他事务插入新行

例外：同一事务内先快照读后当前读，可能看到幻行（因为当前读看最新数据）。严格来说这属于 RR 的一种特殊场景，通过全程用当前读可避免。

Q2：分库分表后如何做全局唯一 ID？跨库分页怎么处理？

A：

  • 全局唯一ID：雪花算法（Snowflake）：时间戳41bit + 机器ID10bit + 序列号12bit，趋势递增，单机每毫秒4096个ID；或 Leaf（美团）、UidGenerator
  • 跨库分页：无法直接 OFFSET，常用方案：
    1. 禁止深翻页，只支持”下一页”（记录上页最后 ID，WHERE id > last_id LIMIT n）
    2. ES 存全量数据做翻页查询，DB 按 ID 查详情
    3. 全局二次排序：各库各取 offset+limit 条，在应用层合并排序取前 limit 条（适合 offset 较小场景）

4.2 ClickHouse（OLAP）

核心知识

核心设计

  • 列式存储：同列数据连续存储，分析查询只读取所需列，大幅减少 IO
  • MergeTree 引擎家族：数据写入时按主键排序，后台异步 Merge 合并文件
    • MergeTree：基础引擎
    • ReplicatedMergeTree：Zookeeper 协调的多副本
    • SummingMergeTree / AggregatingMergeTree：Merge 时自动聚合，适合预聚合场景
    • CollapsingMergeTree：标记删除
  • 向量化执行（SIMD）：列批量运算，充分利用 CPU 向量指令
  • 并行处理：查询自动分配到多核并行，大表扫描性能可线性扩展

写入特点

  • 不支持高频单行写入（每次写入生成 Part 文件，Parts 过多影响查询）
  • 推荐：批量写入（每批 ≥ 1000 行，或间隔 ≥ 1秒），使用 Buffer 引擎或 Kafka 引擎缓冲

与 MySQL 分工

高频面试问答

Q1：ClickHouse 为什么不适合高并发点查？如何解决这个问题？

A：ClickHouse 单次查询会启动大量并发线程扫描数据，对于高并发（如1000 QPS）点查，线程数爆炸导致 CPU/内存耗尽。解决方案：

  • 查询时加 SETTINGS select_sequential_consistency=1 强制一致（性能代价）
  • 写入后查询加入足够延迟（业务可接受的情况下）
  • 对于统计类查询，副本间微小不一致通常可接受

4.4 Elasticsearch

核心知识

核心概念

  • Index → Shard → Segment：索引分为主分片（Primary）和副本分片（Replica），每个分片是独立的 Lucene 索引，由多个 Segment 文件组成
  • 倒排索引：term → [docId, position, …]，全文检索的核心，写入时分词建立倒排表

查询类型

  • match：全文检索，分词后匹配

  • range：范围查询
  • bool：must/should/must_not 组合
  • nested/has_child：嵌套/父子文档查询
  • aggregations：聚合统计（类 GROUP BY）

性能调优

  • 写入优化：批量写（bulk API）；refresh_interval=-1（写入期间关闭自动 refresh）
  • 查询优化：合理设置分片数（避免过多小分片）；使用 Filter 替代 Query（Filter 可缓存）；避免深度分页（用 search_after）

高频面试问答

Q1：ES 分片数量设置的原则是什么？为什么分片数不能修改？

A：分片数在索引创建时确定，之后不可修改（因为文档路由 hash(id) % shard_num 依赖分片数，修改会导致历史文档路由错误）。
设置原则：单分片大小控制在 10-50GB；分片总数 ≤ 节点数 × 3（避免单节点压力过大）；可通过 Rollover（时间序列数据）或 Reindex（重建索引）应对分片调整需求。

6.1 车云 TLS 安全通信 / VAL / T-BOX 协议

核心知识

车云通信安全架构

  • TLS 双向认证（mTLS）：车端（T-BOX）和云端服务器互相验证证书，防止中间人攻击
    • 车端证书：出厂时烧录，或通过 PKI（证书颁发机构）动态签发
    • 证书更新：证书过期前通过 OTA 或专用通道更新
  • 通信协议：MQTT over TLS（端口 8883）/ CoAP over DTLS（UDP，适合低功耗）

T-BOX（Telematics Box）

  • 车辆网联核心硬件，连接 CAN 总线（读取车辆数据：车速/电量/故障码）和蜂窝网络（4G/5G）
  • 数据采集：CAN 报文解析 → 私有协议/MQTT 上报云端
  • 命令下发：云端 → T-BOX → CAN 总线控制车辆（如远程解锁、空调预调）

VAL（Vehicle Abstraction Layer）

  • 车端抽象层，屏蔽不同车型 CAN 矩阵差异，提供统一数据模型
  • 信号定义（Signal Definition）：每个信号对应 CAN ID + 位偏移 + 长度 + 物理量转换公式

安全考虑

  • 消息完整性：HMAC 签名防篡改
  • 防重放攻击：消息携带时间戳 + Nonce，服务端验证时间窗口
  • 证书吊销：OCSP / CRL 检查，防止泄露证书被滥用

高频面试问答

Q1：如何防止车辆远程控制指令被重放攻击？

A：防重放机制：

  1. 时间戳窗口：指令携带时间戳，服务端拒绝超过允许窗口（如30秒）的指令
  2. Nonce（一次性随机数）：每条指令携带唯一 Nonce，服务端 Redis 存储已使用 Nonce（TTL与窗口一致），重复 Nonce 直接拒绝
  3. 指令序号：T-BOX 维护单调递增序号，拒绝小于等于上次序号的指令

6.2 OTA 差分算法（bsdiff/xdelta）

核心知识

OTA（Over-The-Air）升级流程

差分算法原理

  • bsdiff：基于 BWT（Burrows-Wheeler 变换）的二进制差分，生成 patch 文件；bspatch old new patch 还原
    • 优点：压缩率高（差分包通常是全量包的 20-30%）
    • 缺点：生成差分包计算量大，内存消耗高（需将 old 文件全部载入内存）
  • xdelta：流式差分算法，内存占用低，适合大文件；支持 VCDIFF 标准格式
    • 更适合嵌入式设备内存受限场景

OTA 安全与可靠性

  • 完整性校验：SHA-256 校验 patch 文件，防止下载损坏或篡改
  • 签名验证：差分包用私钥签名，设备用内置公钥验证，防止伪造升级包
  • 原子升级：A/B 分区方案，升级写入 Inactive 分区，验证通过后切换 Active，失败自动回滚
  • 断点续传：下载进度持久化，网络恢复后继续下载

高频面试问答

Q1：OTA 升级失败如何保证设备不变砖？

A：可靠性保证通过分层设计：

  1. A/B 双分区：当前运行 A 分区，升级写入 B 分区，升级完成重启切换；启动失败自动回退 A 分区（Bootloader 记录启动尝试次数，超过阈值回退）
  2. 升级前预检：检查存储空间、电量（如 >20%）、网络状态，条件不满足不启动升级
  3. 差分基础版本校验：patch 应用前校验 old 文件 MD5 与差分包声明一致，版本不匹配降级推送全量包

  • Embedding API：文本向量化，用于 RAG 相似性检索

RAG（Retrieval-Augmented Generation）架构

关键指标

  • 召回率（Recall）：相关文档被检索到的比例
  • 精确率（Precision）：检索结果中相关文档的比例
  • 分块策略（Chunking）：影响召回质量，常用固定大小+重叠、按语义边界分块

Prompt 工程

  • 角色设定（System Prompt）、Few-shot 示例、Chain-of-Thought（COT）推理
  • 结构化输出：要求模型返回 JSON，配合 Function Calling 约束输出格式

高频面试问答

Q1：RAG 系统召回效果差如何排查和优化？

A：排查步骤：

  1. 问题诊断：区分”召回了但排名靠后”还是”完全没召回”——前者是排序问题，后者是召回问题
  2. 分块优化：分块太大则语义太宽泛，太小则上下文丢失；尝试父子块（检索小块，返回父块全文）
  3. Embedding 模型：通用 Embedding 在专业领域效果差，考虑领域微调（如 BGE-Finetune）或换用领域相关模型
  4. 混合检索：向量检索（语义）+ BM25（关键词）结合，RRF 重排序，覆盖语义和精确匹配两种需求
  5. Reranker：召回 Top-K 后用 Cross-Encoder 精排（如 BCE-reranker），提升精度

  • Agent：ReAct / Function Calling 驱动，自主调用工具链完成目标

ReAct 框架

高频面试问答

Q1：如何评估 Agent 系统的可靠性？如何防止 Agent 陷入无限循环？

  • 可靠性评估：构建任务测试集，统计任务完成率、步骤数、成本；使用 Ragas 等框架自动化评估

八、编程语言

8.1 Java（主力）

核心知识

JVM 内存模型与 GC

  • 内存区域：堆（新生代 Eden/S0/S1 + 老年代）、方法区（Metaspace）、栈、程序计数器
  • GC 算法：标记-清除（碎片）→ 标记-整理 → 复制算法（新生代 MinorGC）
  • GC 收集器：
    • G1（JDK 9+ 默认）：分区（Region）管理堆，可预测停顿时间（-XX:MaxGCPauseMillis），适合大堆
    • ZGC（JDK 15+）：亚毫秒级停顿，着色指针 + 读屏障，适合超低延迟场景
    • CMS（已废弃）：老年代并发标记，停顿短但产生碎片

Java 并发

  • 线程池（ThreadPoolExecutor）：核心线程数/最大线程数/队列/拒绝策略；避免使用 Executors 工厂方法（隐藏无界队列风险）
  • 锁机制：synchronized（对象监视器）/ ReentrantLock（可中断、公平锁、Condition）/ ReadWriteLock / StampedLock
  • 并发工具：CountDownLatch、CyclicBarrier、Semaphore、CompletableFuture（异步编排）
  • volatile：保证可见性和禁止指令重排，不保证原子性
  • CAS + AQS：Atomic 类的基础；AQS（AbstractQueuedSynchronizer）是 ReentrantLock/CountDownLatch 等的底层框架

Spring 核心

  • IoC：Bean 生命周期（实例化→依赖注入→初始化→使用→销毁）；BeanFactory vs ApplicationContext
  • AOP：JDK 动态代理（接口）/ CGLIB 字节码代理（类）；切面（Aspect）、通知（Advice）、切点（Pointcut）
  • 事务：@Transactional 基于 AOP 代理实现；传播行为（REQUIRED/REQUIRES_NEW/NESTED）；事务失效场景（同类调用自身方法/非 public 方法/异常被捕获）

高频面试问答

Q1：线程池的拒绝策略有哪些？生产上如何避免任务被拒绝？

A：四种内置策略：AbortPolicy（抛异常，默认）、CallerRunsPolicy（调用方线程执行）、DiscardPolicy（静默丢弃）、DiscardOldestPolicy（丢弃最旧任务）。
生产建议：

  • 用 CallerRunsPolicy 作为降级，背压到调用方（调用方变慢，给线程池喘息时间）
  • 监控队列长度，提前告警（队列 >80% 容量时告警），避免被动触发拒绝
  • 核心服务用有界队列（ArrayBlockingQueue）而非无界队列，防止 OOM

Q2：Spring 事务在什么情况下会失效？

A：常见失效场景：

  1. 同类方法自调用：this.methodA() 绕过代理，事务不生效；解决：注入自身代理 bean 或重构为两个类
  2. 非 public 方法：CGLIB 代理无法拦截，注解无效
  3. 异常被 catch 吞掉：事务在 catch 内没有重新抛出，事务不感知异常不回滚
  4. 异常类型不匹配：默认只回滚 RuntimeException，受检异常需 rollbackFor=Exception.class
  5. 多数据源/事务管理器配置错误：@Transactional 用了错误的 transactionManager

8.2 Python（数据分析/自动化）

核心知识

核心数据分析库

  • Pandas：DataFrame 操作，groupby/merge/pivot，处理结构化数据；注意大数据集用 chunking 或 Polars 替代
  • NumPy：高效数组运算，底层 C 实现，向量化操作替代 Python 循环
  • Matplotlib/Seaborn：静态可视化；Plotly：交互式可视化
  • Scikit-learn：ML 模型训练/评估，Pipeline 封装预处理+模型

Python 性能优化

  • GIL（全局解释器锁）：CPU 密集型任务用多进程（multiprocessing）；IO 密集型用多线程或 asyncio
  • asyncio：协程（coroutine）+ 事件循环，单线程处理大量 IO 并发（如批量 API 调用）
  • Cython/Numba：编译加速数值计算；C 扩展（ctypes/cffi）调用底层库

自动化工具

  • paramiko：SSH 自动化运维
  • subprocess：调用系统命令
  • schedule/APScheduler：定时任务

高频面试问答

Q1：Python 的 asyncio 和多线程如何选择？

A：

  • asyncio：适合大量 IO 并发（HTTP 请求、DB 查询、文件读写），单线程无锁，代码逻辑清晰，百万级并发 IO；缺点：同步阻塞调用会阻塞事件循环，需要所有库支持 async
  • 多线程（threading）：适合既有同步库又需要并发的场景，IO 密集型有效（GIL 在 IO 时释放），CPU 密集型无效
  • 多进程（multiprocessing）：CPU 密集型（如图像处理、大数据计算）绕过 GIL

8.3 C++（HPC 高性能计算）

核心知识

高性能编程关键技术

  • 内存管理：RAII（Resource Acquisition Is Initialization）自动管理资源；智能指针（unique_ptr/shared_ptr/weak_ptr）避免内存泄漏；自定义内存池（jemalloc/tcmalloc）减少分配开销
  • 缓存优化：数据结构对齐（Cache Line = 64 Bytes）；AOS vs SOA（Structure of Arrays 更缓存友好）；循环顺序优化（内存连续访问）
  • 并行计算：OpenMP（多核并行）；SIMD（SSE/AVX 向量化）；CUDA（GPU 并行）
  • 编译优化：-O3 -march=native；Link-Time Optimization（LTO）；Profile-Guided Optimization（PGO）

Azure E-HPC 场景

  • MPI（消息传递接口）多节点并行计算
  • InfiniBand 高速互联（低延迟节点间通信）
  • RDMA（远程直接内存访问）绕过 CPU 直接读写远端内存

高频面试问答

Q1：C++ 中如何定位内存泄漏和性能瓶颈？

A：

  • 内存泄漏：Valgrind（memcheck）在 Linux 上检测；AddressSanitizer（-fsanitize=address）编译期注入，运行时检测，性能开销更低
  • 性能瓶颈：perf（Linux 性能事件采样）；gprof（函数级 profiling）；Intel VTune（CPU 指令级分析，缓存未命中/分支预测失误）；FlameGraph 可视化热点调用栈

九、管理与规范

9.1 Scrum 敏捷开发

核心知识

核心角色

  • Product Owner（PO）：维护 Product Backlog，定义优先级，代表业务价值
  • Scrum Master（SM）：消除障碍，守护 Scrum 流程，促进团队持续改进
  • Development Team：自组织，跨功能，通常 3-9 人

核心事件（Sprint = 1-4 周）

  • Sprint Planning：PO + Team 拆解 Sprint Backlog，估算（故事点/理想工时）
  • Daily Standup：15分钟，昨天做了什么/今天做什么/有无阻塞
  • Sprint Review：展示增量，获取 Stakeholder 反馈
  • Sprint Retrospective：团队反思，改进流程（什么做得好/什么需要改进/下一步行动）

度量指标

  • Velocity：每 Sprint 完成的故事点，用于预测能力
  • Burndown Chart：Sprint 内剩余工作量趋势
  • Lead Time / Cycle Time：需求从提出到上线的时间

高频面试问答

Q1：Scrum 在大团队（如 50+ 人）如何扩展？

A：常用**SAFe（Scaled Agile Framework）**或 LeSS（Large-Scale Scrum）：

  • SAFe：引入 ART（Agile Release Train），多团队同步节奏（PI Planning 季度计划会），适合大型企业
  • LeSS：尽量保持 Scrum 简洁，多团队共享一个 Product Backlog 和一个 Sprint，通过 Overall Retrospective 跨团队同步

实践中更多用Spotify 模型：Squad（小队，类似 Scrum Team）→ Tribe（部落，多个 Squad）→ Chapter（章节，同职能人员）→ Guild（公会，跨 Tribe 兴趣社区）

9.2 AI 辅助规范驱动开发（SDD）

核心知识

SDD（Specification-Driven Development）结合 AI

  • 核心理念：先写规范（接口定义、数据模型、业务规则、测试用例），再用 AI 生成实现代码
  • 规范形式：OpenAPI Spec（接口）、JSON Schema（数据模型）、Gherkin（BDD 场景）、架构决策记录（ADR）

AI 辅助开发实践

  • 代码生成：Claude/GitHub Copilot 根据注释、函数签名、规范文档生成实现
  • 测试生成：输入业务规则自动生成单元测试用例（边界条件、异常路径）
  • 代码审查：AI 检查代码规范合规、安全漏洞（SQL 注入、XSS）、性能反模式
  • 文档生成：从代码自动生成 API 文档、架构说明

工程质量保障

  • AI 生成代码必须经人工 Review + CI 测试验证
  • 规范（Spec）由人维护，作为 AI 生成的输入约束
  • AI 作为”高效初稿工具”，工程判断权保留在人

高频面试问答

Q1：AI 辅助开发如何保证代码质量和安全性？

A：

  1. 输入约束：给 AI 明确规范（类型定义、业务规则、不允许的模式），限制生成范围
  2. 强制 Code Review：AI 生成的代码必须经有经验的工程师审查，重点关注业务逻辑正确性、边界条件、安全漏洞
  3. 测试覆盖率门禁：CI 强制要求覆盖率（如 ≥80%），AI 生成的代码同样要通过
  4. 静态分析：SonarQube / Semgrep 扫描 AI 生成代码，检测安全模式
  5. 渐进式引入：AI 先用于低风险模块（工具类、DTO 生成），积累经验后扩展到核心逻辑

9.3 DevOps 全流程

核心知识

CI/CD 流水线

部署策略

  • 滚动更新（Rolling Update）：逐步替换实例，简单，有短暂新旧版本共存
  • 蓝绿部署（Blue-Green）：双环境，流量切换瞬间完成，回滚极快，但成本翻倍
  • 金丝雀发布（Canary）：先放 5% 流量到新版，观察指标正常后逐步放量，精细控制风险

可观测性三支柱

  • Metrics（指标）：Prometheus + Grafana，定量监控系统健康
  • Logs（日志）：ELK/EFK Stack，结构化日志（JSON），集中查询
  • Traces（链路）：SkyWalking / Jaeger，分布式调用链追踪

高频面试问答

Q1：金丝雀发布中如何判断新版本是否健康，何时继续放量？

A：通过自动化指标门禁决策：

  • 错误率：新版本 5xx 错误率 < 0.1%（对比基线）
  • 延迟：P99 延迟无显著回归（如未超过基线 10%）
  • 业务指标：核心业务转化率/下单成功率无下降
  • 资源使用：CPU/内存无异常增长

工具：Argo Rollouts + Prometheus 指标分析自动推进；也可集成 Feature Flag 平台（如 LaunchDarkly）按用户维度精细控制流量分配
