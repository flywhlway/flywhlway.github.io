3. 方案详述（来源：SearXNG代理部署指南）

从你这张图看，SearXNG 本身是正常运行的（至少部分正常），不是“挂了”。

因为它已经能：

  • 正常返回结果页面
  • 至少有一个搜索引擎（google）成功返回结果（0.5s）
  • 聚合结果已经展示出来

所以结论是：

  • SearXNG服务正常
  • 但多引擎出站网络/代理策略存在问题（导致大部分引擎请求超时）

为什么只有 Google 正常，其它都超时？

这通常不是 SearXNG 程序问题，而是 “SearXNG → 代理(7890) → 各搜索引擎” 这条链路里，不同目标站点可达性不一致。

你现在的情况很像下面几种之一（或叠加）：

1）代理规则对不同域名走了不同路径（最常见）

你给 SearXNG 配了 outgoing.proxies 后，所有请求会先到 7890，再由 Mihomo/Clash 按规则决定：

  • Google 可能走到了可用节点 ✅
  • Bing / Brave / DuckDuckGo / Startpage / Wikipedia / Wikidata 可能走了不可用路径（直连或坏节点）❌

这就会表现为：Google 正常，其它引擎超时。

2）代理节点对某些站点质量差/被限流

有些机场节点对 Google 很稳，但对以下站点不稳：

  • bing.com
  • search.brave.com
  • duckduckgo.com
  • startpage.com
  • wikipedia.org
  • wikidata.org

会出现 TCP 能连上但响应慢到超时。

3）你当前搜索词是 URL（https://ai.google.dev），Google更擅长，其他引擎表现差

你搜索的是一个完整 URL。这种查询下：

  • Google 往往很快返回精确命中
  • 其它引擎可能处理慢、命中少，甚至触发超时（尤其在代理不稳定时）

这不是唯一原因，但会放大差异。

4）SearXNG 超时参数偏紧（代理链路下）

如果你配置的是：

  • request_timeout: 6.0
  • max_request_timeout: 15.0

在“代理 + 多引擎 + 不稳定节点”环境里，有些引擎确实会超时。

（你之前还遇到过 extra_proxy_timeout 类型问题，现在修好后建议保留整数并适当加大）

这算“正常”吗？

✅ 算“部分正常”

  • 服务已启动
