3. 方案详述（来源：Docker映射目录权限问题）

你这个现象非常常见：容器里进程默认以 root（UID=0）运行，它在挂载到宿主机的目录里创建的文件自然就变成宿主机上的 root:root，于是 whl 看不到/改不了。

你的目标是：宿主机上由 whl 经常查看、备份、编辑这些挂载目录，同时尽量不破坏容器运行与安全性。下面给你一套“可落地的最佳实践”，按推荐程度从高到低。

结论先说

  • 推荐：让容器用**非 root 用户（固定 UID/GID）**写入挂载目录；宿主机目录也用同一 UID/GID 管理。
  • 可接受：保持 root 写入，但用 ACL 给 whl 永久读写权限（不改 owner）。
  • 不太推荐：直接 chown -R whl:whl 把目录全改成 whl，如果容器仍以 root 或其它 UID 写，后续还会“反复变回 root / 权限打架”，而且可能引入安全隐患。

方案 A（最推荐）：统一 UID/GID，让容器用“whl 同 UID”运行

适用

你能控制 compose，容器镜像支持 user: 或支持 PUID/PGID 这种参数（很多社区镜像都有）。

影响与注意点

  • ✅ 最干净：文件创建出来天然是 whl 可读写，备份/编辑最顺。
  • ✅ 安全性更好：容器不再用 root 写宿主机目录，降低“容器逃逸/误删宿主机”的风险面。

  • ⚠️ 备份工具如果不支持 ACL，可能丢 ACL（用 tar --acls、rsync -A 这类方式即可）。

方案 C（一般不推荐）：直接把目录 chown 给 whl

适用

只有在容器本身也会用 whl 对应 UID/GID 写时才推荐（其实就回到方案 A 的思路了）。

风险

  • 如果容器还是 root 写，文件可能仍然以 root 出现；你又会继续 chown；来回拉扯。
  • 某些应用需要特定 owner（例如它启动时自检目录 owner/权限），你强改 owner 可能导致应用异常或安全机制失效。
  • 如果你把权限放太宽（例如 777），会引入更大安全面（不推荐）。

你这个场景的“最佳实践组合”（建议照这个执行）

  1. 优先尝试方案 A：把服务改为非 root（user: UID:GID 或 PUID/PGID）。
  2. 对少数必须 root 的容器：用 方案 B（ACL） 给 whl 管理权限。
  3. 避免 777；权限控制建议：
    • 数据目录：u+rwX,g+rwX，必要时配合 umask 或应用内权限参数
    • 日志目录：可以只给读权限或分组权限
  4. 统一目录规划（推荐）：
    • /srv/<app>/data
    • /srv/<app>/config
    • /srv/<app>/logs
      并用同一套 owner/ACL 策略管理，后面备份/迁移更省事。
