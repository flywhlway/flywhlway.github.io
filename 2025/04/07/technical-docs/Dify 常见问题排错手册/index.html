技术内容

1. 方案详述（来源：LobeHub 功能调研与部署-dify深度研究DSL）

下面给你一份Dify v1.12.1 / Workflow 模式 / DSL v0.5.0 的企业版工作流 DSL，已将「搜索模块」替换为：

  • ✅ Tavily（默认必跑）
  • ✅ SearXNG（自建备选，配置了就自动参与）
  • ✅ Brave Search API（备选，配置了就自动参与）
  • ✅ 混合搜索聚合：三路结果统一解析 → 合并去重 → 域名黑白名单/每域名上限 → 进入证据流水线

实现方式为：在 Search Round Iteration 内并行/串行调用三路搜索，每个搜索节点都设置 error_strategy: default-value，所以即使你不配置 SearXNG/Brave，也不会影响导入与运行（只会使用 Tavily 的结果）。

✅ 可直接导入 DSL（企业版：Tavily + SearXNG + Brave 混合搜索）

保存为：manus_research_enterprise_tavily_mix_v1_12_1.yml
Dify → Studio → Workflow → Import DSL 导入即可。

导入后在 Environment Variables 填：

  • TAVILY_API_KEY（必填）
  • SEARXNG_BASE（可选，例如 http://searxng:8080）
  • BRAVE_API_KEY（可选）
  • 其他：黑白名单/每域名上限/缓存服务（可选）

若你的 MinIO 域名解析到内网 IP，Dify 的 SSRF 保护可能会阻止访问内网地址；需要按 Dify 文档调整 ssrf_proxy 的 squid 规则 citeturn8search10turn8search6。

2) Gateway 接口建议（就 3 个接口）

  • POST /v1/blob/put-text：写入一段文本（chunk），返回 object_key 与 presigned_get_url
  • POST /v1/blob/compose：把多个 chunk 合成一个最终文件（或写 manifest）
  • GET /v1/blob/presign：按 key 生成临时下载链接（可选）

合成策略：S3/MinIO 不支持真正“追加写”，最简单是 每个 chunk 一个 object，最后在 gateway 里下载拼接再上传一个最终 object（适合报告/markdown）。

3) Dify 内调用：只需要 HTTP Request 节点

Dify 的 HTTP Request 节点就是为了对接外部 API：citeturn0search26

C. Dify 工作流编排：长文“分段生成 → 每段上传 → 最后合并 → 输出引用+摘要”

关键点：不要先生成完整长文再上传

因为：

  • 生成出来会占变量，可能撞 MAX_VARIABLE_SIZE citeturn7view0
  • 上传时 HTTP Request 节点文本体有默认 1MB 限制 HTTP_REQUEST_NODE_MAX_TEXT_SIZE citeturn7view2turn0search9

所以做法是：用 Loop 控制 LLM 分段输出（靠 LLM 节点的 max_tokens 天然控块大小）。

你在 Dify 里按这个顺序搭节点

1) Start

  • 输入：topic、style（可选）

2) Loop 节点

  • 配置 Loop Variables（Loop 变量会跨迭代保留）citeturn2view1turn2view0

    • part：counter，从 1 开始，每次 +1（Dify Loop 示例里就用 counter 自动递增）citeturn2view1turn2view0
    • done：boolean，默认 false
    • summary_md：string，默认空（用于累积摘要）
    • prefix：string，建议初始化为 reports/{{sys.workflow_run_id}}/（系统变量里有 sys.workflow_run_id）citeturn4view0

  • Termination：done == true 或使用 Exit Loop Node（两者任选）citeturn2view1turn2view0

Loop 内部子流程：

2.1) LLM 节点：生成第 N 段

  • max_tokens：例如 1200~2000（你按模型与语种调整）
  • 强制结构化输出（建议 JSON）：
    • chunk_markdown：本段正文（控制长度）
    • chunk_summary：本段 1~2 句摘要
    • is_last：是否最后一段（true/false）
  • Prompt 模板示例：
    • 输入：topic、part、已累积摘要（summary_md）
    • 输出要求：只输出 JSON、chunk 不超过 X tokens、不要重复前文

2.2) HTTP Request 节点：上传本段到 Gateway

  • URL：https://gw.yourdomain.com/v1/blob/put-text
  • Body（JSON）：
    • bucket: dify-artifacts
    • key: {{prefix}}part-{{part}}.md
    • content_type: text/markdown
    • text: {{llm.chunk_markdown}}
  • 输出：拿到 object_key、get_url（预签名下载链接）

如果你想直接调大 HTTP 节点上限：改 HTTP_REQUEST_NODE_MAX_TEXT_SIZE（默认 1048576）citeturn7view2。但更稳仍是分段。

2.3) Variable Assigner：累积摘要
Variable Assigner 支持 overwrite/append 等模式 citeturn1search1

  • summary_md：append - Part {{part}}: {{llm.chunk_summary}}\n

2.4) If/Else：是否最后一段

  • 若 llm.is_last == true：
    • 设置 done = true（Variable Assigner overwrite）
    • 或走 Exit Loop Node

3) Loop 结束后：合成最终文件
再加一个 HTTP Request：

  • POST /v1/blob/compose
  • Body：
    • bucket: dify-artifacts
    • prefix: {{prefix}}
    • dest_key: {{prefix}}report.md
    • part_count: {{part}}（如果 part 是自增 counter，退出时它已是总段数+1/或总段数，按你实现校准一下）

返回：

  • final_object_key
  • final_get_url

4) Output / Answer
输出：

  • 摘要：summary_md
  • 下载链接：final_get_url
  • 以及 final_object_key（方便后续追踪）

目标 2：Iteration/Loop/LLM 尽量分页/分段/分批，并行串行可控

A. 大批量处理的“标准流水线”：外层 Loop 分页，内层 Iteration 处理一页

为什么外层 Loop 分页？

  • 避免把 1000 条塞进一个数组（变量膨胀/调试困难）
  • Iteration 并行模式下，输入数组过大时有已知报错：Max submit count 100 of workflow thread pool reached（典型触发：数组≥101 且并行≥2）citeturn0search3turn0search7

流程结构（可直接照搭）

1) Start

  • 输入：dataset_id、page_size（建议 20~50）

2) Loop（分页游标）
Loop variables：

  • cursor：string，初始为空
  • page：counter，自增
  • done：boolean

Loop 内：

2.1) HTTP Request：拉取一页数据

  • GET https://gw.yourdomain.com/v1/dataset/page?dataset_id=...&cursor={{cursor}}&limit={{page_size}}
  • 返回：
    • items：array（长度 ≤ page_size）
    • next_cursor：string 或 empty
    • has_more：boolean

2.2) If：items 为空或 has_more=false

  • 设置 done=true → Exit Loop

2.3) Iteration：处理这一页 items

  • 输入数组：items
  • 模式：
    • 串行：最稳（慢但几乎不炸）
    • 并行：建议 max_parallelism=3~5（别贪大）
  • 每个 item 内部：
    • LLM/HTTP 做处理
    • 直接把每条结果落存储（再调 Gateway /put-text 或 /put-json）
    • 结果 key 建议：results/{{sys.workflow_run_id}}/page-{{page}}/item-{{index}}.json

Iteration 的输出通常会聚合成数组；如果你不想聚合大数组，就让每条结果“边算边存”，Iteration 的 output 只返回 ok 或一个很短的状态字符串即可（数组再长也很小）。

2.4) 更新 cursor

  • Variable Assigner overwrite：cursor = next_cursor

Loop 结束后：

  • 再 call /compose 把 results/{{run_id}}/ 下的结果合成一个 results.jsonl 或 results.zip
  • 输出下载链接 + 总结

B. 并行/串行控制的实战建议（给你可直接套用的数值）

  1. Iteration 并行：
  • page_size ≤ 50
  • max_parallelism 3~5
  • 若必须处理 >100 条：用分页把每页控制在 50；避免触发 “submit count 100” 类问题 citeturn0search3
  1. Workflow 全局上限（任务多时才需要调）
  • WORKFLOW_MAX_EXECUTION_STEPS、WORKFLOW_MAX_EXECUTION_TIME、MAX_VARIABLE_SIZE 都在 .env.example 里有默认值，按需调大 citeturn7view0
  • Loop 节点可配置上限也受 LOOP_NODE_MAX_COUNT 影响（默认 100）citeturn7view2
  1. HTTP Request 节点 payload 上限
  • 文本 1MB：HTTP_REQUEST_NODE_MAX_TEXT_SIZE
  • 二进制 10MB：HTTP_REQUEST_NODE_MAX_BINARY_SIZE citeturn7view2
    能不调就不调——更推荐分页/分段。

你可以直接照这个“落地检查清单”推进

  • MinIO：bucket + gateway 专用用户最小权限
  • Gateway：写入 chunk、合并、预签名下载
  • Dify：Loop 分段生成（靠 max_tokens 控块）→ 每段 HTTP 上传 → 退出后 compose → 输出链接+摘要
  • 批处理：外层 Loop 分页，内层 Iteration 处理每页；并行控制 3~5；每条结果边算边存

如果你把 你的 Gateway 想要的域名/端口、以及你偏好的实现语言（FastAPI / Node.js）告诉我，我可以直接把：

  • storage-gateway 的完整 docker-compose.yml
  • FastAPI/Node 完整代码（含 MinIO endpoint、预签名 URL、compose 合并、鉴权）
  • 以及对应的 Dify 节点参数映射（每个节点填什么字段）
    一次性给你拼成“复制就能跑”的版本。

7. 方案详述（来源：解除Dify Code节点限制）

下面把 Dify 工作流/节点里最常踩到的“长度/次数/大小/超时”限制，按「节点类型 → 限制内容 → 环境变量/改法 → 更优绕过方案」统一汇总（以官方 docker/.env.example 与 Dify Docs 为准；不同版本变量名/是否生效可能有差异，建议你先对照自己部署目录里的 .env.example）。

0) 先记住：Dify 的限制大致分 3 类

  1. 工作流运行时硬限制：最大步数、最大执行时间、变量最大体积、最大并行分支等（全局影响所有节点）。citeturn4view2
  2. 节点输入/输出校验限制：Code/Template/HTTP Request/Webhook 的 payload/字符串/数组等。citeturn4view2turn3view5
  3. 文件/知识库限制：上传大小、批次数、chunk 附件数、RAG top_k 上限等。citeturn7view2turn7view3turn7view0

1) 工作流运行时（全局）常见限制

2) Code / Template 节点：最常见“输出长度/数组长度”限制

3) Loop / Iteration / Agent：次数/遍历/并发相关限制

4) HTTP Request / Webhook：请求体与响应体大小限制

5) 文件上传 / 知识库 / RAG：大小、批次、top_k 上限

6) LLM 节点“输出字符长度”怎么看？

Dify 本身更常用的是 超时/步数/变量体积 来间接限制“输出字符量”：

  • 单次生成超时：TEXT_GENERATION_TIMEOUT_MS=60000，且官方 troubleshooting 也明确可通过它控制节点运行时长。citeturn4view2turn5view0
  • 输出落到变量会受 MAX_VARIABLE_SIZE 影响（约 200KB）。citeturn4view2

更优方案依旧是：长文本输出落存储（MinIO/DB）→变量里只放引用 + 摘要。

7) 实操建议：你要“统一解决长度限制”，最稳的是这套模式

  1. Dify 只传小变量：task_id / cursor / url / object_key
  2. 重计算/大数组/长文本全部外置（你提到的 OpenAPI + HTTP 节点是很典型的更优解）
  3. 分页是硬通货：Iteration/Loop/LLM 输出都尽量分页或分段
  4. 必要时再调 env：先保守调到“刚够用”，避免把变量/日志/DB 压爆

如果你把你当前版本（docker/.env.example 里那段 runtime 配置）贴出来，或直接贴你 workflow 里是哪几个节点在“超长输出/超大数组/超多循环”，我可以按你实际业务给一套 “Dify 编排 + 外部任务服务(OpenAPI) + MinIO 资产库” 的标准化模板（接口约定、分页协议、Dify 变量映射、错误重试策略）。

8. 方案详述（来源：DifyChat-Docker部署问题排查）

从你这段日志里有个很关键的信号：Prisma migration 能连上 MySQL 并成功执行，但后面“初始化管理员”阶段用连接池拿连接时 10s 超时。这类情况最常见不是密码/URL 写错（否则迁移也会失败），而是：

  1. 平台容器到 MySQL 的网络路径不稳定/被拦（用宿主机 IP 走一圈 NAT/防火墙/策略路由/代理规则），

  3. 容器启动顺序：平台起来时 MySQL 还没完全 ready，迁移重试成功了，但后续初始化管理员的那次连接没拿到（尤其是 MySQL 刚启动后的抖动期、DNS/网络抖动）。
