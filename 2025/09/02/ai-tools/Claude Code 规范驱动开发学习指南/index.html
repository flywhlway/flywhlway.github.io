1. 写在前面：从 Vibe Coding 到 Agentic Engineering

1.1 你为什么读这份指南

如果你已经能熟练地把”想要一个 XX 功能”丢给 AI、再来回调几次拿到能跑的代码，那么 vibe coding 阶段已经过关。但当项目体量超过单文件、超过单个会话、需要多人协作或需要长期维护时，你会发现：

  • AI 经常忽略 CLAUDE.md 中的硬性规则（社区观测约 70% 遵循率）
  • 同一个错误会被反复修复，因为它没有记忆
  • 大型重构容易丢失上下文，丢上下文就丢质量
  • 修一个 bug 引入两个新 bug，没人替你做”验证关卡”
  • 一个会话开得越长，质量越差（上下文窗口超过 40% 后开始降智）

Agentic Engineering（智能体工程）就是为了解决这些工程化问题而生的方法论：把 Claude Code 当作一个团队成员而非聊天框 — 你写规范、给它工具、设置质量门、安排子代理分工，再让它执行。你的角色从”打字员”变成”架构师 + 评审人”。

Boris Cherny（Anthropic Claude Code 负责人）描述过一个数据：Anthropic 工程师每天提交 10–30 个 PR，全部由 Claude 生成；Anthropic 100% 的 PR 在人工评审前先由 Claude 评审一次。这种工作流的杠杆来自工具，更来自方法。

2. Claude Code 核心架构与心智模型

2.1 Claude Code 到底是什么

官方的定义足够精炼：Claude Code 是一个 Agentic Coding 工具，能够读取你的代码库、跨文件编辑、执行命令、与开发工具集成。它有四个形态共享同一套底层能力：

  1. 终端 CLI（最完整、最适合工程化）
  2. VS Code / JetBrains 扩展（inline diff、@-mention、对话历史）
  3. 桌面应用（多会话并行、定时任务、视觉化 diff）
  4. Web 应用 claude.ai/code（云端长任务、不依赖本地环境）

这四个表面共享 同一套 Skills、MCP 连接、Hooks 和会话历史，意味着你在 Mac 终端开始的会话可以在手机上继续。

2.2 心智模型一：Agent + Toolset + Loop

不要把 Claude Code 想成”一个会写代码的聊天机器人”。把它想成：

一个 Agent，它拥有 Shell、文件系统、网络 等工具，在 “读 → 想 → 改 → 验证”的循环 中工作，直到任务完成或你叫停它。

这个循环的关键节点：

理解了这个循环，你才能精准地干预它：CLAUDE.md 干预”读什么”、Skills 干预”用什么方法”、Hooks 干预”允不允许做”、Subagents 干预”由谁来做”。

2.3 心智模型二：六大扩展点

Claude Code 提供了六个独立但可组合的扩展点：

每个扩展点都有最适合它的场景，错配会导致维护成本爆炸（后面第 5 节细讲）。

2.4 心智模型三：上下文是稀缺资源

无论模型多强，上下文窗口都是一种昂贵的稀缺资源：

  • Opus 4.7 拥有 1M token 窗口，但实测在超过 300–400k token 后质量明显下降
  • 一个新会话光是加载系统提示词 + 工具定义 + CLAUDE.md，起步就约 20k token
  • 上下文使用率超过 40% 后开始降智，超过 60% 后明显下降，约 83.5% 时自动压缩（约只保留 20–30% 细节）

这有几个直接的工程含义：

  • CLAUDE.md 必须短而精（参考 60 行上限）
  • 重型探索任务交给 Subagent，因为它有独立窗口
  • 长会话不如多个有焦点的短会话，配合规范文件作为持久化记忆
  • “@-mention 一个大文件”会把整个文件灌进上下文，慎用

4. 核心配置：CLAUDE.md、settings.json 与权限

4.1 配置层级与文件位置

Claude Code 有 三层配置体系，理解层级关系是工程化的基础：

合并优先级：项目级 > 用户级 > 默认。同名 Subagent 也是项目级覆盖用户级。

4.2 CLAUDE.md：项目记忆文件

这是 Claude Code 启动时读的第一个文件。社区共识：

  • 长度上限约 60 行（HumanLayer 团队基准），超过 80 行后 Claude 开始忽略部分内容
  • 写硬性事实和约束，不写性格指令（不要写 “Be a senior engineer”，浪费 Token）
  • 写”为什么”而非”是什么”，要解释清楚反直觉的决策
  • 不在 CLAUDE.md 写 lint/format 规则，那是 Hooks 的活；但如果你没有 formatter，具体的”用 2 空格”是合法的

全局 CLAUDE.md 模板（~/.claude/CLAUDE.md）

项目 CLAUDE.md 模板（Python / FastAPI 后端，参考 Flywhl 的技术栈）

模块级 CLAUDE.md（子目录）

适合大仓库：在 src/auth/CLAUDE.md 放只在进入 auth 目录时才加载的说明。这种”按需上下文”是控制 Token 成本的关键技巧。

4.3 .claude/rules：路径作用域规则

CLAUDE.md 适合写”全项目通用的事实”，但很多规则只对部分文件有效。这时用 .claude/rules/ + paths 通配符：

.claude/rules/python-async.md:

只在编辑 src/services/ 或 src/repositories/ 下的 .py 文件时这个规则才进入上下文。这种”渐进披露”显著减少 Token 占用。

4.4 settings.json：权限与行为控制

~/.claude/settings.json 示例：

权限三档：

  • allow：自动放行，连问都不问（高频安全命令）
  • ask：每次确认（默认对所有 Write/Edit）
  • deny：硬性禁止，连问都不问

attribution.commit: "" 用于关闭 “Co-Authored-By: Claude” 提交签名（如果你不希望 git log 里有它）。这种规则放在 settings.json 比放在 CLAUDE.md 更可靠（100% vs 70%）。

5. 六大扩展机制深度剖析

5.1 Slash Commands：注入预制 Prompt

一句话：/name 把一段预写的 Prompt 模板注入当前上下文。便宜、用户主动触发、共享当前会话。

文件位置：.claude/commands/<name>.md 或 ~/.claude/commands/<name>.md。

示例 .claude/commands/recap.md:

调用：会话中输入 /recap。

适用场景：每天重复做的工作流（PR 摘要、技术债盘点、context dump、发版前检查）。判断标准：如果你一天里把同样的指令重复输入超过 1 次，它就应该是一个命令。

5.2 Skills：渐进披露的领域知识包

一句话：Skills 是一个文件夹，包含 SKILL.md（YAML frontmatter + 指令）和可选的脚本、参考资料。Claude 只看到 name+description（~100 token），只有在判断相关时才加载完整内容（通常 < 5000 token）。

文件位置：.claude/skills/<name>/SKILL.md 或 ~/.claude/skills/<name>/SKILL.md。

最小 Skill 示例

Skills 与 CLAUDE.md 的关键差别

name: code-reviewer
description: Senior code reviewer. Reviews diffs for quality, security, maintainability. Use proactively after any non-trivial code change.
tools: Read, Grep, Glob, Bash
model: sonnet

You are a senior code reviewer with 15+ years of experience.

When invoked:

  1. Run git diff main...HEAD to see what changed.
  2. For each changed file, check:
    • Correctness: Logic errors, off-by-one, race conditions, null/None handling
    • Security: Input validation, SQL injection, secret leakage, auth bypass
    • Maintainability: Naming, duplication, complexity, missing tests
    • Project conventions: Read .claude/rules/*.md and CLAUDE.md, enforce them
  3. Categorize issues:
    • 🔴 Blocking: Must fix before merge
    • 🟡 Suggested: Improve if cheap
    • 🟢 Nit: Style/minor

Output a markdown review with code excerpts and concrete suggestions. Don’t write code yourself — your job is to find issues, not fix them.

If everything is clean, say so. Don’t manufacture criticism.

7.2 第一天：Specify

Claude 会问你：错误率怎么定义？阈值多少？回滚的”目标版本”怎么选？决策需要谁审批？审计日志写到哪？……一问一答 20 分钟后产出 spec.md。

7.3 第二天：Plan + 评审

拿到 plan.md（假设包含 8 个 task），打开编辑器手动标注：

  • Task 2 的事件总线写错了，我们用 Kafka 不是 RabbitMQ
  • Task 4 漏了多租户隔离
  • Task 7 的”通知”应该走我们的统一告警平台，参考 services/alerting/

迭代 2–3 轮后，plan.md 进入”可执行”状态。Commit 它。

7.4 第三天：Implement（用 Subagent 并行）

每个 Subagent 拿到的输入：plan.md 的本任务 + 相关代码上下文。它们产出代码 + 测试 + 一句话总结。

中间随时 /agents 查看正在运行的子代理状态。

7.5 第四天：Validate

7.6 这个工作流为什么有效

  • 决策一次性收敛：Specify 阶段把模糊问题问清，避免实施期间反复返工
  • 质量门把关：Plan 评审挡住低质量计划，validate 挡住未达标实施
  • 上下文清洁：Subagent 隔离实施细节，主代理上下文不被污染
  • 规范即文档：spec.md / plan.md / validation.md 直接就是项目文档
  • 可重入：任何一天你新开会话，加载 specs 文件夹立刻接上下文

8. 推荐安装的高品质开源插件与技能

筛选标准：万星以上、近期活跃、有清晰的安装路径、能直接装到 Claude Code。

8.1 安装 Plugin 的通用流程

Claude Code 通过 Plugin Marketplace 分发插件。每个 marketplace 是一个 GitHub 仓库。

8.2 必装第一梯队

① anthropics/skills（37.5k+ ⭐）— 官方技能集合

Anthropic 官方的技能仓库，是理解 Skill 设计的最佳样板。包含：

  • document-skills：docx、pdf、pptx、xlsx 的创建/编辑/读取（你做技术文档时极有用）
  • example-skills：各种领域示例 — algorithmic-art、canvas-design、mcp-builder、artifacts-builder 等

② obra/superpowers（社区基石）— TDD/调试/规划方法论

obra/superpowers 是社区最受认可的方法论插件，把”软件工程文化”打包成 14 个 SKILL.md：

  • brainstorming：开始写代码前用 Socratic 提问澄清需求
  • writing-plans：把特性拆成 2–5 分钟的可测试任务
  • test-driven-development：强制 RED → GREEN → REFACTOR，禁止”以后补测试”
  • systematic-debugging：四阶段调试法，三次失败后强制架构评审
  • subagent-driven-development：派遣子代理实施 + 第二个子代理评审
  • writing-skills：用 TDD 方法写新 Skill

实测体感：装上后第一个反应是”Claude 变慢了” — 因为它现在每次都会先 brainstorm/plan。但产出质量肉眼可见的提升，整体项目周期反而缩短 2–3 倍。

③ wshobson/agents（35.6k+ ⭐）— 多智能体编排市场

79 个聚焦插件、184 个专业 Agent、150 个 Skill，按场景模块化。最强的”按需安装”市场。

热门组合：

每个插件只加载自己的 Agent/Skill 到上下文（约 1000 token 量级），不会膨胀。

④ anthropics/claude-plugins-official（官方目录）

Anthropic 官方维护的高质量插件目录，含 Cloudflare、Pinecone、PlanetScale、Prisma、Pydantic AI、Qdrant、Airflow 等厂商官方插件。

8.3 第二梯队（按场景挑）

文档与上下文

  • Context7 MCP（必装基础）：版本特定的库文档
  • upstash/context7 系列：处理 Redis/Upstash 时增强
  • claude-code-handbook 类：社区 Prompt Engineering 手册

浏览器与前端

  • Microsoft Playwright MCP：浏览器自动化（万星级）
  • shadcn/ui 插件：UI 组件生成（如果用 React）

数据与数据库

  • PostgreSQL MCP（官方）
  • Supabase MCP（如果用 Supabase）
  • Pinecone / Qdrant / Weaviate MCP：向量数据库（贴合你做 RAG/嵌入应用）

Vehicle IoT / 嵌入式（贴合 Flywhl 兴趣）

  • arm-cortex-microcontrollers（wshobson）：STM32/nRF52/SAMD/Teensy 固件开发
  • embedded-systems：嵌入式系统模式
  • blockchain-web3（如果涉及车联网数据链上存证）

AI 工程（贴合 Flywhl 的”具身智能”目标）

  • pydantic-ai（官方）
  • langgraph / crewai 相关 Skill 仓库
  • mcp-builder（来自 anthropics/skills）：写自己的 MCP 服务器

8.4 安装策略与建议

  1. 从最小集开始：anthropics/skills + obra/superpowers + Context7 MCP + GitHub MCP，覆盖 80% 场景
  2. 按项目按需加：每个项目 .claude/settings.json 里只启用相关插件，避免污染其它项目
  3. 每月做一次”卸载审计”：装而不用的插件徒增 Token 成本
  4. 来源可信优先：装 anthropic、microsoft、官方厂商的，社区插件只装高星活跃的
  5. 本地试跑再加固定：新插件先 /plugin install 试一周，确认有用再写入团队配置

8.5 自己写一个 Plugin / Skill

如果某个工作流你做了三次以上，又找不到现成插件，把它写成 Skill。

最小 Skill 结构：

写好后，直接放到团队的 Plugin marketplace 仓库就能分享给整个团队。建议团队建一个 <org>-claude-plugins 仓库作为内部市场，所有共识写在那里。

9. 上下文工程与成本控制

9.1 监控上下文使用

会话中输入 /cost 看实时 Token 消耗，输入 /context 看上下文占用比例。40% 是预警线，60% 是行动线。

9.2 上下文卫生的几条原则

  1. 新会话 vs 长会话：复杂任务到一半，与其继续，不如 dump → /clear → resume
  2. 使用 dump 模式：让 Claude 把当前进度写到 markdown，然后 /clear，新会话读那个 markdown
  3. 避免 @-mention 大文件：@docs/api-spec.md 会把整个文件灌进去；改成 “如果遇到 X 问题，读 docs/api-spec.md”
  4. 重型探索用 Subagent：探索操作放进子代理隔离窗口
  5. 关闭不用的 MCP：Playwright MCP 不用时关掉，单次会议化能省 100k+ token

10. 常见反模式与排错指南

10.1 反模式：CLAUDE.md 是个垃圾桶

症状：CLAUDE.md 长到 300 行，Claude 还是不遵守

根因：你把所有项目特性都堆进去了

修法：

  1. 拆成 .claude/rules/*.md + paths 通配符
  2. 拆成子目录 CLAUDE.md
  3. 把”做法”挪进 Skills
  4. 主 CLAUDE.md 控制在 60 行内

10.2 反模式：让 Claude 当保姆带

症状：每修一个 bug 都要给详细步骤”先看 X 文件然后修改 Y”

根因：你没相信它的探索能力，反而限制了它

修法：贴 bug 描述、说 “fix”，别教它怎么修

10.3 反模式：跳过 plan 直接 implement

症状：实施 30 分钟后才发现方向错了

根因：模糊决策没收敛

修法：强制走 --plan 模式，妥协一次也不行

10.4 反模式：长会话不分段

症状：上下文超过 80%，Claude 开始忘记早期的指令

根因：你舍不得 /clear

修法：到 60% 就 dump + clear + resume

10.5 反模式：在多步规划中阻止 Write

症状：Hook 中途阻塞了 Write，Claude 后续思路混乱

根因：你以为 PreToolUse 是安全网，但它打断了推理

修法：把”质量门”放在 PostToolUse 或 Stop，不阻 Write

10.6 反模式：太多 MCP

症状：每次回答前思考很久，工具选择困难

根因：5+ MCP 服务器，工具列表里几十个工具

修法：精简到 3–5 个高频用的，其它按需启动

11. 进阶路线图与持续学习

11.1 学习里程碑（按 30/60/90 天）

第一个 30 天 — 熟练度：

  • 安装、认证、IDE 集成全跑通
  • 至少 3 个项目有自己的 CLAUDE.md
  • 用过 --plan 模式做过 1 个完整特性
  • 配置过自己的 Slash Commands 与 Hooks
  • 装上 anthropics/skills + obra/superpowers + Context7 MCP

第二个 30 天 — 工程化：

  • 写过至少 2 个自己的 Skill
  • 写过至少 1 个自己的 Subagent
  • 在某个项目走过完整的 spec → plan → implement → validate 流程
  • 配过 PostToolUse + Stop hook 做自动测试/lint
  • 至少装过 3 个 MCP 服务器

第三个 30 天 — 多代理与扩展：

  • 用 Subagent 并行跑过研究任务
  • 用 Agent Teams 模式协作完成一个大特性
  • 把团队规范沉淀成 <org>-claude-plugins 仓库
  • 用 Claude Code SDK 写过一个自动化脚本（CI 集成、批量重构）
  • 能给同事讲清”Skills vs Subagents vs Hooks”的边界

11.2 持续学习的几个源

  • 官方文档：https://docs.claude.com/en/docs/claude-code/overview — 唯一权威
  • Anthropic 博客：模型与功能发布
  • Anthropic Skills 仓库：github.com/anthropics/skills — 学习 Skill 写法
  • awesome-claude-code（hesreallyhim）：精选资源
  • wshobson/agents：观察”产品级 Agent 编排”长什么样
  • obra/superpowers README：方法论参考
  • alexop.dev：阅读社区实战派的工作流文章
  • 每月扫一次 Changelog：Claude Code 迭代非常快（2026 年 3–4 月两个月发了 30+ 版本）

11.3 与 Flywhl 当前职业目标的对接建议

针对 IoT 云平台架构师 + 具身智能方向的求职准备，Claude Code 可以作为：

  1. 简历与面试准备的协同伙伴：让 Claude 反向面试你（用 AskUserQuestion），把每个项目逼到具体数字、决策、tradeoff
  2. 架构图与文档生成器：装 c4-architecture 类插件，用 mermaid/PlantUML 自动生成图
  3. 技术原型沙箱：把 XOTA / Galileo / E-ACE 三个项目的核心场景重写成可演示的 mini repo
  4. 具身智能学习项目：跑一个本地 LLM + 机器人仿真的 demo，全用 Claude Code 驱动开发，写成 blog
  5. 每周技术博客：把工作中的技术沉淀整理成长文（你已经在 vault 里做了，但流程可以更工程化）

12.5 工程化检查清单

新项目接入 Claude Code 前的 10 分钟自检：

  • CLAUDE.md 在仓库根目录，< 60 行
  • .claude/ 目录已建，settings.json 设了权限
  • 至少装了 Context7 MCP
  • 至少有 1 个 .claude/rules/*.md 用于路径作用域
  • CLAUDE.local.md 在 .gitignore
  • PostToolUse hook 跑 lint/format
  • Stop hook 跑 fast tests
  • 有 /recap、/handoff 命令
  • 有 code-reviewer subagent
  • docs/specs/ 目录已建，有 spec.md 模板
  • 团队 README 引用了一个”如何加入开发”段落（如何用 Claude Code 开始）

写在最后

Claude Code 的真正杠杆不在模型本身（虽然 Opus 4.7 已经很强），而在围绕模型搭建的工程化外壳：CLAUDE.md 给它项目知识，Skills 给它”如何做”的方法论，Hooks 给它纪律，Subagents 给它专业分工，MCP 给它外部世界，规范驱动开发给它清晰的目标。

“The skill of directing, evaluating, and building with AI tools is already one of the most valuable capabilities a professional can hold.”

这种能力的价值随 AI 变强而上升，不是下降。把现在花在掌握 Claude Code 上的时间，看作对未来几年生产力的复利投资。

祝你从 vibe coder 成长为 agentic engineer 的旅途顺利。
