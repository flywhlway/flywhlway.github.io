1. 前言：为什么 Cursor 值得被认真学习

1.1 从 “vibe coding” 到 “spec-driven coding”

如果你已经具备一定的 vibe coding 经验，应该体会过这种循环：打开 chat，凭直觉写 prompt，看 AI 输出，不满意就再聊几轮，最后把能跑的代码 commit 进去。这种方式对小工具、原型、个人项目极其高效，但放到中大型项目里几乎一定会失败：

  • 上下文会爆——单轮聊天很快塞满 token，模型开始忘掉前面的约定；
  • 代码风格会漂移——同一个项目里，三天前生成的代码用 Tailwind，三天后又冒出 styled-components；
  • 架构会失控——你以为只是加了个接口，AI 顺手新建了三个 service 类和一个反模式的循环依赖；
  • review 成本反超手写——表面写得快，PR 一打开发现要重构一半。

2026 年的工程团队对 AI 编码工具的期望已经从”快”转向了”稳”，关键词是 规范驱动开发（Spec-Driven Development, SDD）：先有清晰的规范、规则、计划，再让 AI 作为执行体在受约束的边界内工作。Cursor 是当前最适合落地这种工作流的工具之一——不是因为它的模型最强（模型层 Cursor 是路由，Claude Opus / GPT-5.5 / Composer 2 / DeepSeek 都能选），而是因为它在 IDE 层提供了：

  1. Rules / Skills / Commands 三层可复用上下文机制
  2. Plan 模式 + Agent 模式的”先计划后执行”分离
  3. Agents Window + /worktree + /best-of-n 的并行/隔离执行
  4. MCP 原生支持让 AI 能够查数据库、跑 API、读 Figma
  5. Cloud Agents 把长任务从本地剥离，可异步交付 PR

如果你已经熟悉 vibe coding，那么这份指南的真正价值在于：教会你何时不该 vibe coding，以及当项目复杂度超过临界点时，如何用一套受控、可审计、团队可复用的工作流去交付严肃软件。

1.2 这份指南的承诺与边界

这份指南会做的事：

  • 完整覆盖 Cursor 3.x（截至 2026 年 5 月）所有核心特性；
  • 提供可复制粘贴的配置、规则、命令、MCP 服务器示例；
  • 给出从入门到熟练的渐进式学习路径；
  • 用真实项目场景（IoT 云平台、连接车 OTA 子模块）做端到端演示；
  • 同时讲解 Cursor + Codex、Cursor + Claude Code 两套主流协作工作流；
  • 标注每个开源资源的 GitHub 星标量与适用性。

  • 不替你做技术选型决策（前端用 React 还是 Vue，由你的团队决定）；

2. Cursor 全景认知：2026 年的产品形态

2.1 Cursor 是什么、不是什么

Cursor 是：一个 AI-native 的 IDE，由 Anysphere 出品，从 VS Code fork 而来，但在 AI 集成层做了深度重写。所有 VS Code 的扩展、键位、主题、设置同步几乎无缝迁移。

Cursor 不是：一个 VS Code 插件。这点至关重要——它不是把 GitHub Copilot 换成了 Claude，而是把 AI 当成了 IDE 的一等公民，索引、上下文管理、agent 调度都在 IDE 内部直接处理。

到 2026 年中，Cursor 的用户超过 100 万开发者，付费用户约 36 万，进入了 Fortune 500 中 64% 的企业，并在 2025 年成为有史以来最快达到 1 亿美元 ARR 的 SaaS 公司之一。它的产品定位是清晰的：承担从单行补全到端到端特性交付的全谱 AI 编码场景。

3. 入门配置：从零搭建专业开发环境

目标：从下载 Cursor 到第一个 commit，30 分钟内完成一个生产可用的开发环境。

3.1 系统准备

Cursor 3.x 的系统要求：

  • macOS：12 Monterey+（Apple Silicon M1/M2/M3/M4 原生支持）
  • Windows：10 1903+（ARM 预览支持）
  • Linux：Ubuntu 20.04+、Debian 10+、Fedora 36+、Arch 等主流发行版
  • 磁盘：至少 4GB 给应用本体，索引另算（中型项目 1~3 GB）
  • 网络：所有 AI 调用都在云端，需要稳定外网

对于本指南的目标读者（Mac M5 Pro 48GB 这类配置），完全无需担心硬件——本地 Composer 2 推理也跑得很轻松。

3.2 安装与初始登录

首次启动：

  1. 选择登录方式：GitHub、Google、邮箱（推荐 GitHub，方便后续 Cloud Agent 接入仓库）；
  2. 选择是否导入 VS Code 配置（强烈建议勾选——所有扩展、键位、设置一键迁移）；
  3. 选择主题；
  4. 进入欢迎页。

3.3 必做的 8 项初始设置

打开 Cursor Settings（Cmd+, 或左下齿轮），按顺序完成：

① 启用 Privacy Mode（强烈建议）

Settings → Privacy Mode，打开后 Cursor 承诺零数据保留、不会用你的代码训练模型。Teams 计划默认强制启用。

② 配置默认模型

Settings → Models：

  • Default Chat Model：Auto（推荐）或 Claude Sonnet 4.7
  • Default Agent Model：Auto 或 Composer 2
  • Premium Models：开启 Claude Opus 4.7、GPT-5.5、Composer 2

③ 启用 Codebase Indexing

Settings → Indexing & Docs：

  • 开启 Codebase Indexing；
  • 等待状态栏出现 “Indexed”（取决于项目大小，5 秒到几分钟不等）；
  • 索引完成后，Cursor 才能在 chat 里全局搜索代码。

④ 配置 .cursorignore（创建模板）

在你的项目根目录创建 .cursorignore：

⑤ 配置 Auto Update

Settings → Application → Auto Update：保持开启。Cursor 迭代速度极快，过一周不更新就会落后。

⑥ 启用 Workspace Trust

Settings → Workspace Trust：从 Cursor 3 起，Workspace Trust 默认关闭——务必启用。它会在你打开陌生仓库时阻止恶意 .vscode/settings.json 或 MCP 自动加载。CVE-2025-64106 就是 deep-link RCE 漏洞，启用 Workspace Trust 是基线防御。

⑦ 配置 Shell Integration

Settings → Terminal：

  • Default Terminal：你日常用的（zsh / bash / pwsh）；
  • Enable shell integration：开启（让 agent 能正确解析命令输出）；
  • 把你的 .zshrc / .bashrc 中的关键环境变量也加进 Settings → Agent → Environment Variables（agent 子进程不一定继承交互 shell 的环境）。

⑧ 启用 “Made with Cursor” 归属（团队场景可关）

Settings → Agent → Attribution：是否在 commit message 里附加 “Made with Cursor” 注释。个人偏好，企业版可以集中禁用。

3.4 第一次完整工作流体验

跑一次完整流程，确认环境就绪：

  1. 新建测试项目：

  1. 用 Cursor 打开：cursor .

  2. 等待索引完成：状态栏出现 “Indexed”。

  3. 试 Tab 补全：在 README.md 里输入 ## Featu，Cursor 应该补全 ## Features 并继续推断段落。

  4. 试 Inline Edit：选中 README.md 的标题行，按 Cmd+K，输入 “make it more enthusiastic”，看 diff。

  5. 试 Ask 模式：Cmd+L 打开 chat panel，切到 Ask，问 “How is this project structured?”。

  6. 试 Agent 模式：切到 Agent，让它 “Create a Python FastAPI hello-world skeleton with uvicorn”。观察它如何创建文件、运行命令、生成 requirements。

  7. 试 Plan 模式：切到 Plan，让它 “Plan a small REST API with 3 endpoints: /health, /items GET, /items POST”。审阅生成的 Markdown 计划，然后转 Agent 执行。

如果以上 8 步都顺畅完成，你的 Cursor 环境就绪了。

3.5 推荐的扩展兼容性检查

Cursor 兼容大部分 VS Code 扩展，但有些 AI 类扩展会冲突。建议禁用或卸载：

  • ❌ GitHub Copilot（与 Cursor Tab 冲突）
  • ❌ Codeium（同上）
  • ❌ Continue.dev（如果不专门用，关掉避免 token 浪费）

保留并优化：

  • ✅ GitLens（Git 历史可视化）
  • ✅ Error Lens（行内错误提示，加速 review）
  • ✅ Prettier / ESLint（代码格式化）
  • ✅ Docker / Kubernetes 扩展（Cursor 的 agent 能识别它们的语义）
  • ✅ Material Icon Theme（看着舒服）
  • ✅ Path Intellisense（路径自动完成）

4. 核心交互模型：Tab、Inline Edit、Ask、Agent、Plan

官方建议的学习路径：先掌握 Tab → 然后 Inline Edit → 再 Ask → 最后 Agent / Plan。直接跳到 Agent 的开发者最容易”过度信任 AI”，错过 bug。

4.1 Tab：智能下一步预测

Tab 不是简单的下一行补全。它是 next-edit prediction——Cursor 会综合当前光标位置、周边代码、最近编辑历史，预测你”接下来要做的整个编辑动作”，可能是：

  • 在当前位置插入几行；
  • 跳到另一处（同文件甚至跨文件）继续相关修改；
  • 提示一次 multi-line 重命名/重构。

示例场景：你给一个 TypeScript interface 加了一个新字段 email: string。按下 Tab 后，Cursor 可能会：

  1. 跳到该 interface 使用的几个 mock 数据处，提示给每个对象加 email: 'xxx@example.com'；
  2. 跳到对应的 Zod schema，提示加 email: z.string().email()；
  3. 跳到表单组件，提示加 <input name="email" />。

使用技巧：

  • 接受：Tab
  • 拒绝：Esc
  • 逐字接受：Cmd+→ (macOS)
  • 切到上一个建议：Alt+[

最佳实践：

  • Tab 适合惯性继续，不适合让它跨越意图边界。如果你正在写认证逻辑，但 Tab 跳到了一个数据库 schema 里改字段，多半是它误判了你的意图，按 Esc。
  • 不要为了用 Tab 而硬接受不合适的建议。Tab 的最大成本不是按下错的代码，而是后续 review 它写出的隐性错误。

4.2 Inline Edit (Cmd+K)：外科手术式改写

选中一段代码 → Cmd+K → 输入自然语言指令 → Cursor 返回 colour-coded diff → 接受/拒绝/部分接受。

适合场景：

  • 单函数重构（”Convert this to async/await”）
  • 添加错误处理（”Add try-catch with structured logging”）
  • 类型标注补全（”Add proper TypeScript types”）
  • 单元测试生成（”Generate unit tests for this function using vitest”）
  • 注释生成（”Add JSDoc comments”）

反面案例：用 Inline Edit 让它”重写整个文件并加入 5 个新功能”——这超出了 Inline Edit 的设计目的，应该用 Agent。

5. 上下文工程：@-mention、索引与 .cursorignore

核心命题：AI 编码工具的真实瓶颈不是模型，是上下文。哪些代码、文档、数据被塞进 prompt、以什么顺序、占多少 token——决定了输出质量上限。

5.1 索引：Cursor 怎么知道你的项目

打开项目时，Cursor 会做：

  1. AST 解析关键语言（TS/JS/Python/Go/Rust/Java 等）；
  2. 本地生成 embedding（句子级别，文件路径加密）；
  3. embedding 上传 Cursor 后端，但代码原文不上传也不持久化；
  4. 检索：用户提问时，把语义相关片段拉回作为上下文。

索引何时失效 / 慢：

  • 仓库 >10GB 或文件数 >50万：索引会非常慢；
  • monorepo 没配 .cursorignore：node_modules 会被索引吃掉所有时间；
  • 频繁切分支：增量索引，但偶尔会触发全量。

实操建议：

  • 大 monorepo 强制配 .cursorignore；
  • 项目 IO 慢时检查 Settings → Indexing 看进度和当前路径；
  • 定期 “Resync Index”（同菜单）解决偶发不一致。

6. Rules 系统：从 .cursorrules 到 .cursor/rules/*.mdc

核心思想：Rules 是给 AI 的持久化系统提示。每次新 chat、每次 agent 调用，都会自动注入。把团队约定、架构原则、命名规范、安全要求写进 Rules，是中大型项目使用 Cursor 的第一步基础工程。

6.1 Rules 的三个层级

description: One-line summary explaining what this rule does (used by AI to decide when to apply)
globs: src//*.ts, src//*.tsx
alwaysApply: false

Backend API Conventions

Error Handling

  • ALWAYS throw from src/errors/index.ts ApiError class
  • NEVER swallow errors silently; if intentional, comment // intentional swallow: <reason>
  • Use structured logging via import { logger } from '@/lib/logger'

API Response Format

  • Success: { ok: true, data: ... }
  • Error: { ok: false, error: { code: string, message: string } }
  • Pagination: { ok: true, data: [...], pagination: { cursor, hasMore } }

Database Queries

  • Use Drizzle ORM, never raw SQL outside src/db/migrations
  • ALWAYS use transactions for multi-table writes
  • Add created_at, updated_at to every table

.cursor/rules/10-typescript.mdc

  • 真正的异常（不可恢复）才用 throw

异步

  • 优先 async/await，禁止链式 .then()
  • 并发用 Promise.all，注意短路 fail-fast 语义
  • 避免裸 await 在循环里（除非确实要串行）

6.5 Rules 的反模式

写了几个月 Rules 后我们整理出的常见坑：

❌ 反模式 1：把规范当规则写

这是规范，不是规则。规范该放 docs/，规则该放 .cursor/rules/。规则是给 AI 的指令，规范是给人的契约。混淆会导致：

  • 规范变化时忘了改规则；
  • 规则文件臃肿到 AI 读不完；
  • 团队 review 规范走 PR 流程，改规则却没人审。

❌ 反模式 2：超长 alwaysApply 文件

alwaysApply: true 的文件会注入每次 prompt。如果它有 2000 行，每次对话 token 池都先少 2000 行。alwaysApply 文件加总应控制在 500 行以内。

6.6 Rules 的演进：从 0 到团队级

  • 第 0 天：复制本文 §6.4 的模板，根据项目改字段；
  • 第 1 周：每次 review 发现 AI 写错了团队规范，把规范固化到 .mdc；
  • 第 2 周：拆分 .cursorrules 为多个 .mdc，按文件类型 / 模块 glob；
  • 第 1 个月：定期（每两周）audit Rules，删过时的、合并重复的；
  • 第 2 个月：建立”Rules CR”流程——修改 Rules 走 PR，至少一个 reviewer。

7. Commands、Skills 与 Subagents：可复用自动化的三件套

2026 年 Cursor 演化出的三个相邻概念，初次接触容易混淆。一句话区分：Rules 引导（被动），Skills 执行（主动），Commands 触发（快捷）。

7.1 Commands：可复用的 prompt 快捷键

Commands 是保存为 Markdown 的 prompt 模板，放在 .cursor/commands/[name].md，在 chat 输入框打 / 触发。

典型 Commands 集合（来自社区高星仓库 hamzafer/cursor-commands 与本指南整理）：

.cursor/commands/code-review.md

调用方式：在 chat 里输入 /code-review，Cursor 会把这段 prompt 直接展开。

.cursor/commands/create-pr.md

.cursor/commands/generate-migration.md

Commands 的优势：

  • 不占 alwaysApply 的 token；
  • 团队共享（commit 进仓库）；
  • 拼接灵活：/code-review + 选中代码 = 即时 review；

8. MCP 生态：把 Cursor 接入你的真实工具链

MCP = Model Context Protocol，Anthropic 提出、各家工具厂商响应的开放标准，让 AI agent 通过统一协议接入外部工具/数据。Cursor 2.x 起原生支持，3.0 起有 MCP Marketplace。

8.1 MCP 是什么、为什么重要

传统插件机制是：每个 IDE 各自定义扩展 API，工具厂商要写 N 套适配。MCP 标准化了：

  • Tools：可调用的函数（如 query_database、create_issue）；
  • Resources：可读取的数据（如 schema、文档片段）；
  • Prompts：可注入的提示模板。

Cursor 作为 host，启动 MCP server（本地进程或远程 SSE），agent 自动发现这些 server 暴露的 tools，按需调用。

对工程师的真实价值：

  • 数据库：让 agent 直接查 Postgres，不需要你 copy-paste schema；
  • GitHub：agent 看 issue、提 PR、读评论，不用切窗口；
  • Slack：agent 给团队发部署通知；
  • Figma：agent 读设计稿的实际颜色值、间距；
  • 内部 API：agent 调用你的 staging 环境验证假设。

Cursor 默认每次 MCP 工具调用都需要你确认。这是好事。

  • Manual：每次都问（最安全，最慢）
  • Auto-approved list：你信任的 server 自动批准

实战建议：

  • 本地 filesystem、git → auto-approve；
  • 数据库查询 → manual approve（防止 agent 误删数据）；
  • 写操作（提 PR、删文件、发消息）→ 永远 manual；
  • 生产环境的 MCP → 用 read-only 权限的凭证。

10. 规范驱动开发（SDD）：让 Cursor 严肃服务于中大型项目

10.1 vibe coding 在中大型项目为什么必败

随项目规模增大，单纯的 chat-driven 开发会遇到：

  1. 回归不可控：AI 改 A 时不知道 B 也依赖这块；
  2. 架构腐化：每次”快速实现”积累技术债；
  3. 知识丢失：决策只在 chat 历史里，离职即失传；
  4. review 不可信：reviewer 不知道 AI 当时为什么这么写；
  5. 测试覆盖虚高：AI 写的测试经常只是镜像实现（恒真）；
  6. 责任模糊：bug 出来，”AI 写的” 不是合规借口。

SDD 的核心主张：把”AI 应该做什么” 变成可读、可审、可版本控制的规范文档。代码是规范的实现，不是规范的替代品。

10.4 Cursor 在 SDD 中的”局限性”

要诚实：Cursor 是 SDD 的执行环境，不是SDD 系统。它有这些天然不足（来自社区与 Augment Code 的分析）：

  1. 没有 spec 生命周期管理：spec 改了，老代码是否还合规？Cursor 不会自动追踪。
  2. Rules 不等于 spec：Rules 是 AI 提示，spec 是工程契约。混淆会导致审计真空。
  3. 会话不延续：跨 session 的 spec 上下文不自动加载（需要每次 @ 引用）。
  4. 没有自动一致性校验：实现偏离 spec 时不会主动报警。

补偿方案：

  • 把关键 spec 条款（如”决策必须 deterministic”）也固化到 .cursor/rules/，让 AI 在每次生成时被提醒；
  • 在 CI 里跑”spec compliance”检查脚本（可用 LLM 自检：把 spec + diff 给 Claude，问”是否合规”）；
  • 用 SDD 专用工具（如 OpenSpec、Intent、Kiro）做更严格的生命周期管理，Cursor 做执行层。

15. 反模式与陷阱：那些没人会告诉你的坑

我们整理了 30+ 个真实踩过的坑，分类如下。这些经验只有踩过才会知道，建议收藏并定期回看。

15.1 Prompt 反模式

❌ “改一下” / “修一下” / “优化一下”

模糊指令的输出永远是赌博。AI 会按它对”改”的猜测去做，可能是改名、可能是 refactor、可能是删一半。

改进：写明 “what / why / how-to-verify”：

❌ “尽量…” / “如果可以…” / “可能的话…”

LLM 会把”尽量”理解为”能省就省”，结果就是省。

改进：明确强度：

❌ 一次性塞 10 件事

“加这个接口、改那个 schema、补这几个测试、顺便把日志格式化下、对了 README 也更新一下”。最后某 1~2 件被遗忘，且不报。

改进：拆成 1~3 件一组，分批让 agent 做，每批 review 后再下一批。

❌ 不给样例

“写一个数据校验函数”——AI 不知道你团队风格，可能用 yup、joi、zod、valibot、手写。

改进：给”模仿对象”：

15.2 Cursor 配置反模式

❌ 不配 .cursorignore

索引慢、token 浪费、敏感文件可能被读。

改进：从 §3.3 模板开始，按项目调整。

❌ alwaysApply: true 文件超过 3 个或加总超过 500 行

每次 prompt 注入大量规则，token 池快速消耗，且 AI 反而注意力涣散。

改进：精挑细选 alwaysApply 内容，其余按 globs 自动 attach。

❌ 不区分 User Rules 和 Project Rules

个人偏好（”用中文回复”）写进 Project Rules，让团队都中招。

改进：个人偏好 → User Rules（settings 内）；团队约定 → .cursor/rules/。

❌ Rules 文件互相矛盾

两条规则都 alwaysApply：一条说”使用 var”，一条说”使用 const”——AI 行为完全不可预测。

改进：定期 audit Rules（每两周）、删除/合并冲突项。

15.3 Agent 使用反模式

❌ 不审就跑（YOLO 模式 in production repo）

agent 执行 rm -rf node_modules && git clean -fd，连同你 stash 的 wip 一起没了。

改进：YOLO 只在沙箱、可重置环境；生产 repo 永远 manual approve。

❌ Long-running session

agent 跑了 80 轮还在 chat 同一会话。后半段它已经忘了开头的约束。

改进：50 轮内必须 /summarize 或开新 chat；把对齐内容固化到 Rules。

❌ 同时让 Cursor agent 和 Claude Code 改同一个文件

git 救不了你。

改进：一文件一时刻一 agent，必要时分 worktree。

❌ 信任 agent 的”测试通过”声明

agent 跑了 pnpm test 看到 “All passed”——但它跑的是 cached 结果或只跑了 1 个文件。

改进：永远自己再跑一遍最终验证。

15.4 MCP 反模式

❌ 给 agent 写权限的生产数据库 MCP

有 prompt injection 风险（包括来自 AI 误判），agent 删库不是段子。

改进：生产 DB MCP 用 readonly 凭证；写操作走人工 confirm。

❌ 装 20 个 MCP server

超过 40 工具上限，部分被截断；context 过载，agent 决策质量下降。

改进：精选 3~5 个真正常用的；其他需要时手动启用。

❌ 直接用从未 review 的社区 MCP server

CVE-2025-54136 的教训——MCP server 可能被恶意修改命令。

改进：用前看 source、看作者、看 star/issue 历史；偏好官方 + 知名社区。

15.5 SDD 实施反模式

❌ 写 spec 但不维护

spec 是”半年前需求评审遗留物”，代码早改了。

改进：spec 改动走 PR；merge 前必须双更新（spec + code）。

❌ spec 太详细到等于代码

spec 写到伪代码级别，实际上等于先用文字写一遍代码，再让 AI 翻译——浪费。

改进：spec 描述”what + why + invariant”，代码描述”how”。

❌ Rules 和 spec 重复

同一约束在 spec、Rules、CLAUDE.md 各写一遍，改起来要改三处，必漏一处。

改进：单一源 + 引用。

❌ 不做 ADR

半年后看代码：为什么是这么设计的？没人记得。

改进：每个架构级决策一个 ADR，几句话就够。

15.6 模型选择反模式

❌ 任何任务都用 Opus

信用池一周烧光。

改进：日常 Auto；只在真正复杂任务用 Opus。

❌ 永远不切模型

即使任务明显适合 GPT-5.5 / Composer 2，也用 Claude Sonnet。

改进：知道每个模型的强项（§2.3），根据任务切。

❌ 用便宜模型做架构设计

省钱，但生成的设计漏洞百出。

改进：架构、设计、关键算法 → 用最强模型（不省这点钱）；实施、测试、文档 → 用普通模型。

15.7 团队协作反模式

❌ 每个开发者各自维护 Rules

A 的 Rules 和 B 的 Rules 完全不同，code style 漂移。

改进：Rules 入仓库、走 review 流程，集中维护。

❌ 不培训

新人看到 Cursor 直接 Agent 模式发飙，写出一堆问题代码。

改进：内部 wiki 写明 Tab → Inline Edit → Ask → Agent 的渐进路径，新人先用前两个。

❌ 不集中管理 MCP 凭证

每个开发者本地一份 token，离职时没人记得 revoke。

改进：团队部署 MCP Gateway（如 TrueFoundry），凭证集中。

❌ 不监控 AI 工具成本

月底突然 $3000 账单，没人能解释。

改进：team admin 在 Cursor / Anthropic / OpenAI 控制台都开 budget alert；每月 review 用量分布。

15.8 容易被忽视的”隐性”陷阱

⚠️ Cursor 索引 vs Git 不一致

你切了分支，索引可能还在旧分支状态，agent 引用了不存在的文件。

改进：切分支后看状态栏，”Indexing…” 完成再开始 agent 任务。

⚠️ 多 workspace 时 .cursor/rules/ 加载顺序

打开 monorepo 时哪个 workspace 的 rules 优先？文档不太清楚。

改进：核心规则放仓库根的 .cursor/rules/，sub-package 的 rules 放各 package 内并明确标范围。

⚠️ Tab 接受了不该接受的代码

Tab 提示太快，你下意识 Tab 一下接受了 30 行错误代码。

改进：心理上把 Tab 当”建议”而非”自动完成”，每次接受前快速扫一眼 diff。

⚠️ Cursor 自动更新打断工作

正在 agent 任务时 Cursor 自动重启更新。

改进：关键任务期间临时关 auto update；或安排在固定时间更新。

⚠️ “Done!” 后没说改了哪些 file

agent 报告完成，但具体改了什么没列。后续看 git diff 才发现有意外改动。

改进：prompt 里明确要求 “List all modified files at the end.”；review 时永远 git status + git diff。

16. 团队协同、安全与成本治理

单人开发 Cursor 已经够爽，但要推广到团队，有一套额外的工作要做。这一章面向 tech lead / 平台架构师。

16.1 团队上车路径

Step 1：私下试点（1~2 周）

  • 1~2 个 senior eng 个人订阅 Pro；
  • 在内部小项目 / 边缘项目用，收集体验；
  • 写一份内部”Cursor 入门 vs 我们项目”指南。

Step 2：小团队试用（2~4 周）

  • 整队（5~10 人）开 Teams 计划试用；
  • 协同制定 .cursor/rules/ 和 .cursor/commands/；
  • 衡量 PR 速度 / 代码质量变化（用 DORA metrics + bug rate）；
  • 收集每个工程师的痛点和省时点。

Step 3：组织级铺开（1~2 个月）

  • 申请 Enterprise（如有 SSO/SCIM/审计需求）；
  • 强制 Privacy Mode；
  • 集中部署 MCP Gateway；
  • 培训：办内部 workshop，渐进 Tab → Inline → Ask → Agent；
  • 制定使用规范（哪些代码不该用 AI、谁有权用 Cloud Agent、cost cap 等）。

Step 4：持续治理

  • 月度 Rules audit；
  • 季度成本回顾；
  • 工程师定期分享高效用法。

16.2 安全治理清单

数据安全

  • ☐ Privacy Mode：所有团队成员强制启用（Teams/Enterprise 默认）；
  • ☐ 不让 Cursor 索引 secrets/、.env*、*.pem、*.key；
  • ☐ MCP 凭证最小权限原则（数据库 readonly、token 限定 scope）；
  • ☐ Workspace Trust 永远开启；
  • ☐ 重要仓库的 Rules 文件改动必须 PR 审；
  • ☐ AI 生成的代码必须人工 review 才能 merge。

凭证管理

  • ☐ 团队部署 MCP Gateway 集中托管 token；
  • ☐ 使用 1Password CLI / macOS Keychain 注入而非 .env；
  • ☐ 季度 rotate token；
  • ☐ Cursor 账号 SSO 接入企业 IdP。

合规

  • ☐ 列入 AI 工具治理清单（用于 SOC2/ISO27001 audit）；
  • ☐ 与法务确认数据出境/隐私合规（中国/欧盟特别注意）；
  • ☐ Cloud Agent 跑业务数据时确认 SLA 和数据驻留区域。

已知 CVE 跟踪

  • ☐ 订阅 Cursor changelog + GitHub Security Advisories；
  • ☐ 安装/MCP 来源白名单（不允许任意社区 npm package）；
  • ☐ 重要事件后及时升级。

React 规范

组件

  • 优先函数组件 + hooks，禁止 class 组件（旧代码不强制改）
  • 文件一个默认导出 + 命名导出（如 types）
  • props 接口名 <ComponentName>Props，使用 type 别名

State 管理

  • 局部状态：useState
  • 跨组件共享：Context API + useReducer（小规模）/ Zustand（中等）
  • 服务端状态：TanStack Query (React Query) v5
  • 表单状态：react-hook-form + zod resolver

Effect

  • useEffect 必须列全部依赖；ESLint exhaustive-deps 不可关闭
  • 优先在 event handler 而非 useEffect 中触发副作用
  • 数据获取使用 useQuery，禁止 fetch 在 useEffect 中

性能

  • memo / useMemo / useCallback 只在性能 profile 后使用，不为”防御性”
  • 长列表使用 react-window 或 react-virtuoso
  • React 18 的 useTransition / useDeferredValue 用于非紧急更新

样式

  • Tailwind CSS 为主
  • 复杂组件用 CVA (class-variance-authority) 管理变体
  • 禁止内联 style 除非动态值

  22. 15. 反模式与陷阱：那些没人会告诉你的坑
    1. 15.1 Prompt 反模式
      1. ❌ “改一下” / “修一下” / “优化一下”
      2. ❌ “尽量…” / “如果可以…” / “可能的话…”
      3. ❌ 一次性塞 10 件事
      4. ❌ 不给样例
    2. 15.2 Cursor 配置反模式
      1. ❌ 不配 .cursorignore
      2. ❌ alwaysApply: true 文件超过 3 个或加总超过 500 行
      3. ❌ 不区分 User Rules 和 Project Rules
      4. ❌ Rules 文件互相矛盾
    3. 15.3 Agent 使用反模式
      1. ❌ 不审就跑（YOLO 模式 in production repo）
      2. ❌ Long-running session
      3. ❌ 同时让 Cursor agent 和 Claude Code 改同一个文件
      4. ❌ 信任 agent 的”测试通过”声明
    4. 15.4 MCP 反模式
      1. ❌ 给 agent 写权限的生产数据库 MCP
      2. ❌ 装 20 个 MCP server
      3. ❌ 直接用从未 review 的社区 MCP server
    5. 15.5 SDD 实施反模式
      1. ❌ 写 spec 但不维护
      2. ❌ spec 太详细到等于代码
      3. ❌ Rules 和 spec 重复
      4. ❌ 不做 ADR
    6. 15.6 模型选择反模式
      1. ❌ 任何任务都用 Opus
      2. ❌ 永远不切模型
      3. ❌ 用便宜模型做架构设计
    7. 15.7 团队协作反模式
      1. ❌ 每个开发者各自维护 Rules
      2. ❌ 不培训
      3. ❌ 不集中管理 MCP 凭证
      4. ❌ 不监控 AI 工具成本
    8. 15.8 容易被忽视的”隐性”陷阱
      1. ⚠️ Cursor 索引 vs Git 不一致
      2. ⚠️ 多 workspace 时 .cursor/rules/ 加载顺序
      3. ⚠️ Tab 接受了不该接受的代码
      4. ⚠️ Cursor 自动更新打断工作
      5. ⚠️ “Done!” 后没说改了哪些 file
