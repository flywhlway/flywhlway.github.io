2. 方案详述（来源：Surfshark与mihomo对比）

明白：你的 *.flywhl.vip 不是指向你服务器公网 IP，而是 CNAME → cpolar 公网域名 → 再回到服务器内网服务。
这种情况下，如果服务器/容器自己去访问 https://xxx.flywhl.vip，就会发生典型“发夹绕圈”（出站走 Surfshark → 访问 cpolar 域名在海外/公网 → 再穿隧道回到自己），会带来延迟、慢、偶发不通。

所以最终方案要改成：

  • 出站兜底：Surfshark WireGuard + kill-switch（不变）
  • 本机访问自家域名：强制解析到内网/本机（split-horizon，避免绕 cpolar）
  • mihomo：用 proxy-providers 自带订阅更新 + 机场优先、不可用回落 DIRECT（= Surfshark）
  • DNS：对 *.flywhl.vip 做 fake-ip 例外，避免解析被 fake-ip 干扰

1) Surfshark VPN vs mihomo+机场：核心差异对比

A. 稳定性与可控性

  • Surfshark（商业 VPN）
    • 优点：节点供给更可控、抗封锁能力通常更稳定（同一运营商/地区时波动更小）、账号体系更规范。
    • 缺点：你对“线路细节/中转/策略”掌控更少；有时会对数据中心/协议做限制；个别地区也可能被针对。
  • mihomo + 机场订阅
    • 优点：玩法多（分流/规则/策略组/负载均衡/故障切换/延迟选优）、成本低、线路多样。
    • 缺点：机场本质是“多源拼装线路”，同一订阅在某一时间段可能“全灭”（你已经遇到），并且节点质量与维护靠机场方，波动不可预测。

结论：如果目标是“长期稳定可用”，Surfshark 更适合当“底座出口”；机场更适合当“补充/加速/特定站点优化”。

    3. 断网保护（kill-switch）（避免 VPN 掉线时直连泄漏）
